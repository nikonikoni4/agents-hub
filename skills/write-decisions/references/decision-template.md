# 决策文档模板

创建决策文档时使用此模板。保留与当前决策有关的内容，删除无内容的占位说明。

```markdown
---
version: 1.0
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
last_updated: YYYY-MM-DD
abstract: 一句话说明这个决策解决什么问题，以及最终选择或当前状态
status: decided
---

# 决策标题

## 版本

| 版本 | 更新内容 |
| ---- | -------- |
| 1.0 | 创建文档初稿 |

## 问题界定

### 问题简述

说明为什么现在必须做这个决策。写清触发决策的真实问题，不要只复述用户想做什么。

### 讨论范围

- 本次决策覆盖的模块、流程、接口、数据、用户体验或工程边界。
- 本次决策需要回答的关键问题。

### 非讨论范围

- 明确排除的内容。
- 容易被误认为相关、但本次不处理的问题。

### 模糊信息的明确定义

- `{模糊词或概念}`：在本文中具体指 `{定义}`。

若没有模糊信息，删除本节。

### 问题深度

说明这是浅层方案选择，还是涉及架构原则、产品策略、长期维护方式、协作机制等更深层问题。

## 现状

描述当前系统、流程或决策环境。必须写出现有问题、约束、已知事实和不能忽略的风险。

## 可选方案

### 方案 A：{方案名}

说明方案内容。

**优势**

- 优势 1
- 优势 2

**劣势**

- 劣势 1
- 劣势 2

### 方案 B：{方案名}

说明方案内容。

**优势**

- 优势 1
- 优势 2

**劣势**

- 劣势 1
- 劣势 2

## 最终决策

选择 `{方案名}`。

如果当前没有做出决策，将 frontmatter 的 `status` 改为 `undecided`，并删除本节。

## 决策原因

- 原因 1：说明该原因如何对应问题界定和现状约束。
- 原因 2：说明为什么它比其他方案更适合当前上下文。

## 后续影响

- 对代码结构、数据、接口、测试、文档、运维或用户体验的影响。
- 需要后续验证或复盘的事项。
```
