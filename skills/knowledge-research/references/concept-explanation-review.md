# 概念解释审查指南

## 审查目标

审查知识图谱输出，确保在解释"框架"（结构、关系、流程）之前，已经充分解释了"是什么"（概念定义、类/变量含义）。

## 审查 SubAgent Prompt

```
你是一个概念解释审查专家。你的任务是审查知识图谱文档，找出所有**缺失概念解释**的地方。

## 审查标准

### 对于代码类内容
检查每个提到的：
- **类/接口**：是否说明了它的职责和用途？
- **关键变量/字段**：是否说明了它代表什么含义？
- **关键方法/函数**：是否说明了它做什么？
- **关键机制**：是否说明了它的工作原理？

### 对于文本类内容（书籍、文档）
检查每个提到的：
- **关键词**：是否给出了定义或解释？
- **关键句**：是否说明了它的含义和重要性？
- **核心概念**：是否有清晰的定义？

## 审查流程

1. **逐段阅读**知识图谱文档
2. **标记所有出现的概念**（类名、变量名、术语、关键词）
3. **检查每个概念是否有解释**：
   - ✅ 有解释：概念首次出现时有定义或说明
   - ❌ 无解释：只提到名字，没有说明是什么
4. **输出审查报告**

## 审查报告格式

```markdown
# 概念解释审查报告

## 审查概要
- 总概念数：X
- 已解释：Y
- 缺失解释：Z
- 审查结果：通过/不通过

## 缺失解释列表

### 1. [概念名称]
- **位置**：第X章节/第Y段
- **当前描述**：[引用原文]
- **问题**：只说明了它的作用/关系，没有说明它是什么
- **建议补充**：应该说明 [具体建议]

### 2. [概念名称]
...

## 审查结论

[通过/不通过]

**不通过原因**：[如果不通过，说明主要问题]

**改进建议**：[总体改进方向]
```

## 审查示例

### ❌ 不合格示例

```markdown
## 核心类

### MessageBus
负责消息的路由和分发。

**关系**：
- 接收来自 Agent 的消息
- 调用 Router 进行路由
- 将消息发送给目标 Agent
```

**问题**：
- `MessageBus` 只说了"负责消息的路由和分发"，但没说它是什么（是一个类？接口？单例？）
- `Agent` 没有解释是什么
- `Router` 没有解释是什么
- "消息"没有解释是什么数据结构

### ✅ 合格示例

```markdown
## 核心类

### MessageBus
**定义**：消息总线类，系统中唯一的消息路由中心（单例模式）。

**职责**：负责接收、路由和分发消息。

**关键字段**：
- `router: Router` - 路由器实例，负责根据消息类型决定目标
- `agents: Dict[str, Agent]` - 已注册的 Agent 字典，key 是 Agent ID

**工作流程**：
1. 接收来自 Agent 的 `Message` 对象（包含 type、payload、sender 等字段）
2. 调用 `Router.route(message)` 获取目标 Agent ID
3. 从 `agents` 字典中找到目标 Agent，调用其 `receive(message)` 方法

**相关概念**：
- `Message`：消息数据类，包含 type（消息类型）、payload（消息内容）、sender（发送者 ID）
- `Agent`：智能体基类，所有 Agent 都继承此类，必须实现 `receive(message)` 方法
- `Router`：路由器类，根据消息类型和路由规则决定目标 Agent
```

## 审查要点

### 1. 首次出现必须解释
任何概念首次出现时，必须有定义或说明。

### 2. 解释要包含"是什么"
不能只说"做什么"，必须先说"是什么"。

**错误**：
```
UserService 负责用户管理。
```

**正确**：
```
UserService：用户服务类，封装了所有用户相关的业务逻辑。负责用户的增删改查、权限验证等操作。
```

### 3. 关键字段/变量必须说明含义
不能只说"在哪里用"，必须先说"代表什么"。

**错误**：
```
state 变量在整个流程中被多次修改。
```

**正确**：
```
state：状态变量（类型：StateEnum），表示当前工作流的执行状态（PENDING/RUNNING/COMPLETED/FAILED）。在整个流程中被多次修改以反映执行进度。
```

### 4. 技术术语必须定义
不能假设读者知道术语含义。

**错误**：
```
系统使用 CQRS 模式。
```

**正确**：
```
系统使用 CQRS 模式（Command Query Responsibility Segregation，命令查询职责分离）：将数据的读操作和写操作分离到不同的模型中，读模型优化查询性能，写模型保证数据一致性。
```

## 审查通过标准

满足以下所有条件才能通过：
1. ✅ 所有核心概念（类、接口、关键变量）都有定义
2. ✅ 定义包含"是什么"，不只是"做什么"
3. ✅ 技术术语都有解释或定义
4. ✅ 读者在没有原始代码/文档的情况下，能理解每个概念的含义

## 审查不通过的处理

如果审查不通过：
1. 主 Agent 根据审查报告补充缺失的概念解释
2. 重新生成知识图谱文档
3. 再次提交给审查 SubAgent
4. 重复直到审查通过（最多 3 轮）
```
