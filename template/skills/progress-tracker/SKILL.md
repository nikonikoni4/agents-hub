# Progress Tracker Skill

收集零散想法，整理成结构化任务，追踪进度并生成报告。

## 存储结构

```
docs/
├── temp/
│   └── ideas/              # 用户随手写的地方（原始输入）
└── progress/               # AI 整理后的地方
    ├── ideas.md            # Ideas 收集箱
    ├── tasks.md            # 任务列表
    └── reports/            # 定期报告
        └── YYYY-MM-DD.md
```

**首次使用时**：如果 `docs/progress/` 不存在，自动创建。

---

## 触发场景

| 场景 | 触发词 | 动作 |
|------|--------|------|
| 被动收集 | "整理 ideas"、"收集想法" | 扫描 docs/temp/ideas/ → 提取 → ideas.md |
| 主动输入 | "记一下..."、"我有个想法..." | 直接记录到 ideas.md |
| 确认想法 | "看看 ideas"、"哪些可以做" | 展示待确认 → 转化为任务 |
| 进度报告 | "总结进展"、"进度报告" | 读取 tasks.md → 生成报告 |

---

## 阶段 1：收集（Collect）

### 1.1 被动收集：扫描 docs/temp/ideas/

**步骤**：
1. 读取 `docs/temp/ideas/` 下所有 `.md` 文件
2. 区分用户编写 vs AI 编写（见下方判断逻辑）
3. 提取想法内容
4. 追加到 `docs/progress/ideas.md`

**用户 vs AI 内容判断**：

| 特征 | 用户编写 | AI 编写 |
|------|----------|---------|
| 格式 | 松散、不统一 | 整齐、有结构 |
| 内容 | 零散、跳跃 | 完整、有逻辑 |
| 标记 | 少用 emoji/checkbox | 大量使用 |
| 长度 | 短句、碎片化 | 长段落 |

**处理方式**：
- 用户内容：原样保留，不转换格式
- AI 内容：提取结构化信息

### 1.2 主动输入：用户直接说

**触发**：
- 用户说"记一下"、"我有个想法"、"想法是..."
- 用户描述一个想法或需求

**步骤**：
1. 提取用户的想法内容
2. 追加到 `docs/progress/ideas.md`
3. 确认记录成功

---

## 阶段 2：确认（Confirm）

### 2.1 展示待确认的想法

**触发**："看看 ideas"、"哪些可以做"、"确认想法"

**步骤**：
1. 读取 `docs/progress/ideas.md`
2. 筛选状态为 `💭 待确认` 的想法
3. 展示列表，等待用户确认

**展示格式**：

```markdown
## 待确认的想法

### 1. [日期] <想法标题>
- **内容**：<摘要>
- **类型**：<类型>

### 2. [日期] <想法标题>
...

---
请告诉我：
- 哪些可以转化为任务？
- 哪些需要进一步讨论？
- 哪些可以废弃？
```

### 2.2 处理确认结果

根据用户反馈：

| 决定 | 操作 |
|------|------|
| 可以做 | 状态改为 ✅，创建任务到 tasks.md |
| 需要讨论 | 状态改为 ❓，记录讨论点 |
| 暂时搁置 | 保持 💭，添加备注 |
| 废弃 | 状态改为 ❌，记录原因 |

---

## 阶段 3：结构化（Organize）

### 3.1 创建任务

**从已确认的想法创建任务**：

使用 TaskCreate 工具创建任务，同时更新 tasks.md。

**tasks.md 格式**：

```markdown
# 任务列表

## 进行中

### [任务标题]
- **来源**：idea #[N] / 用户直接指定
- **状态**：🚧 进行中
- **创建时间**：YYYY-MM-DD
- **描述**：<任务描述>

## 待开始

### [任务标题]
- **来源**：idea #[N]
- **状态**：⏳ 待开始
- **创建时间**：YYYY-MM-DD
- **描述**：<任务描述>

## 已完成

### [任务标题]
- **来源**：idea #[N]
- **状态**：✅ 已完成
- **完成时间**：YYYY-MM-DD
- **描述**：<任务描述>
```

### 3.2 生成进度报告

**触发**："总结进展"、"进度报告"

**步骤**：
1. 读取 tasks.md
2. 统计各状态任务数量
3. 生成报告写入 `docs/progress/reports/YYYY-MM-DD.md`

**报告格式**：

```markdown
# 进度报告

**日期**：YYYY-MM-DD
**统计周期**：<起始日期> ~ <结束日期>

---

## 概览

| 状态 | 数量 |
|------|------|
| ✅ 已完成 | N |
| 🚧 进行中 | N |
| ⏳ 待开始 | N |
| 💭 想法待确认 | N |

---

## 已完成

1. <任务标题> - <简要说明>
2. ...

## 进行中

1. <任务标题> - <当前状态>
2. ...

## 待开始

1. <任务标题> - <等待条件>
2. ...

---

## 近期想法

- <最近收集的想法摘要>

---

## 建议下一步

1. <优先级最高的待办事项>
2. <可以开始的新任务>
```

## Ideas.md 格式

```markdown
# Ideas 收集箱

> 此文件由 progress-tracker 自动维护，请勿手动编辑

---

## 待确认

### 💭 YYYY-MM-DD <想法标题>
- **来源**：<文件路径> / 用户直接输入
- **内容**：<原始内容或摘要>
- **类型**：灵感 / 观察 / 需求 / 问题

---

## 已确认（可转化为任务）

### ✅ YYYY-MM-DD <想法标题>
- **内容**：<内容>
- **决定**：转化为任务
- **任务**：见 tasks.md #N

---

## 需要讨论

### ❓ YYYY-MM-DD <想法标题>
- **内容**：<内容>
- **讨论点**：<需要讨论的问题>

---

## 已废弃

### ❌ YYYY-MM-DD <想法标题>
- **原因**：<为什么废弃>
```

---

## 重要约束

1. **不强制结构化**：ideas.md 保持轻量，只在转化为任务时才结构化
2. **尊重原始内容**：用户编写的内容原样保留，不转换格式
3. **确认再转化**：想法必须经过用户确认才能转化为任务
4. **定期清理**：提醒用户定期回顾 ideas.md，避免积压
5. **报告自动生成**：进度报告基于 tasks.md 自动生成，不依赖手动更新
