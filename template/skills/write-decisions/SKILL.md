name: write-decisions
description: Use when 用户做出重大决策（架构调整、模块重构、新功能添加、旧功能删除）或用户出现决策疑问（不知道如何解决某个设计问题和非知识问答问题）时，需要记录具体决策内容
---

## 执行流程
<execute-flow>
1. 按照下面的 check-list 逐项确认，确保你已经完整捕捉到用户的核心问题：
   1. 确认不在讨论范围之外的内容
   2. 确认当前讨论范围之内的内容
   3. 确认问题中涉及到的关键信息明确清晰，无歧义，不模糊
   4. 确认用户问题的深度，是就当前问题解决方案的浅层讨论，还是涉及到更深层次的原理问题讨论
2. 确保当前决策满足一下三个条件才创建：
   1. 难以逆转​ —— 将来改变主意的成本是可观的
   2. 脱离上下文会令人惊讶​ —— 未来的读者会疑惑"他们为什么这样做？"
   3. 真实权衡的结果​ —— 存在真正的替代方案，并且你出于特定原因选择了其中一个
2. 编写决策文档

说明：这里所说的“问题”，解释为：

- 对于重大决策，问题是指，为什么要进行架构调整，当前架构存在什么问题；为什么要进行模块重构？；为什么要添加这个新功能？为什么要删除旧功能？
- 对于用户决策疑问，问题直接指用户的决策疑问：比如某个 tool 工具设计有多种不同的方案，该如何选择？
</execute-flow>

## 决策文档编写

### frontmatter

```yaml
version:
created_at:
updated_at:
last_updated:
abstract:
status:
```

status：`decided` 或 `undecided`

### 文档正文

1. 版本

   ```
      | 版本 | 更新内容 |
      | ---- | -------- |
      | 1.0 | 创建文档初稿 |
   ```

2. 问题界定：写明

   - 问题简述

   - 讨论范围

   - 非讨论范围

   - 模糊信息的明确定义（若都明确，这部分内容可省去）

   - 问题深度

3. 现状（当前的问题）

4. 可选方案（如果有多种方案，需要都写出来）以及可选方案的优势和劣势

5. 最终决策和决策原因（如果没有决策，则不写）

**写入位置**：`docs/design-decisions`

**命名规则**：YYYY-MM-DD-{具体内容}.md

**编写完成之后**：在`docs/design-decisions/index.md`增加导航内容

### 决策文档模板

创建 `docs/design-decisions/YYYY-MM-DD-{具体内容}.md` 时使用下面模板。保留与当前决策有关的内容，删除无内容的占位说明。

```markdown
---
version: 1.0
created_at: YYYY-MM-DD
updated_at: YYYY-MM-DD
last_updated: YYYY-MM-DD
abstract: 一句话说明这个决策解决什么问题，以及最终选择或当前状态
status: decided
---

# 决策标题

## 版本

| 版本 | 更新内容 |
| ---- | -------- |
| 1.0 | 创建文档初稿 |

## 问题界定

### 问题简述

说明为什么现在必须做这个决策。写清触发决策的真实问题，不要只复述用户想做什么。

### 讨论范围

- 本次决策覆盖的模块、流程、接口、数据、用户体验或工程边界。
- 本次决策需要回答的关键问题。

### 非讨论范围

- 明确排除的内容。
- 容易被误认为相关、但本次不处理的问题。

### 模糊信息的明确定义

- `{模糊词或概念}`：在本文中具体指 `{定义}`。

若没有模糊信息，删除本节。

### 问题深度

说明这是浅层方案选择，还是涉及架构原则、产品策略、长期维护方式、协作机制等更深层问题。

## 现状

描述当前系统、流程或决策环境。必须写出现有问题、约束、已知事实和不能忽略的风险。

## 可选方案

### 方案 A：{方案名}

说明方案内容。

**优势**

- 优势 1
- 优势 2

**劣势**

- 劣势 1
- 劣势 2

### 方案 B：{方案名}

说明方案内容。

**优势**

- 优势 1
- 优势 2

**劣势**

- 劣势 1
- 劣势 2

## 最终决策

选择 `{方案名}`。

如果当前没有做出决策，将 frontmatter 的 `status` 改为 `undecided`，并删除本节。

## 决策原因

- 原因 1：说明该原因如何对应问题界定和现状约束。
- 原因 2：说明为什么它比其他方案更适合当前上下文。

## 后续影响

- 对代码结构、数据、接口、测试、文档、运维或用户体验的影响。
- 需要后续验证或复盘的事项。
```

## 决策总结编写
### user-design-summary.md 模板

如果 `docs/design-decisions/user-design-summary.md` 不存在，创建该文件并使用下面模板；如果已存在，在对应章节追加观察。

```markdown
# User Design Summary

## 目的

记录用户在重大设计决策中的偏好、判断方式和风险倾向，为后续 agent 自主决策提供参考，也为用户复盘决策方式提供材料。

## 决策行为观察

| 日期 | 关联决策 | 观察 | 可参考性 | 风险提醒 |
| ---- | -------- | ---- | -------- | -------- |
| YYYY-MM-DD | [决策标题](./YYYY-MM-DD-{具体内容}.md) | 用户在本次决策中体现出的判断方式 | 后续 agent 可参考的条件 | 不应盲目复用的边界或风险 |

## 决策偏好总结

- 当前可观察到的稳定偏好。
- 偏好成立的上下文条件。
- 可能导致误判的倾向。

## 给后续 agent 的参考

- 可以参考的判断原则。
- 必须重新确认的前提。
- 不建议直接沿用的做法。
```

## 决策总结
1. 路径：`docs/design-decisions/user-design-summary.md`
2. 文档说明：这个文档是用来记录用户决策的行为方式，是用来积累用户的决策行为，为agent自主决策时提供依据，以及为用户决策反思提供过程
3. 编写原则：编写时以批判性视角进行说明，不要刻意迎合用户观点，需要产生用户决策进化，以及提供agent可参考的决策思维。如果你认为不太可取的决策，需要明确说明，让后续agent不要盲目参考该决策方式
