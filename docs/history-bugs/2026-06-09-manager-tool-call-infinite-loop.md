# Manager 工具调用无限循环

**创建时间**: 2026-06-09  
**状态**: 已识别  
**严重程度**: 高（浪费上下文空间，堵塞消息传递）

## 问题描述

Manager agent 在处理任务时陷入无限循环，反复执行相同的工具调用（TodoWrite 和 Read），导致上下文空间被大量重复内容占用。

### 表现症状

- TodoWrite 被调用 50+ 次，内容完全相同
- 同一个文件（`docs/temp/调研结果.md`）被读取 30+ 次
- 每次 Read 返回 `Wasted call — file unchanged since your last Read`
- 模型不断说"让我先更新 todo list，然后制定详细的方案"，但从未进入实现阶段

### 循环模式

```
1. Thinking: "让我制定 OpenCode 的集成方案"
2. TodoWrite: 更新进度（相同内容）
3. Read: 读取调研结果.md（文件未变化）
4. Thinking: "让我制定 OpenCode 的集成方案"
5. 回到步骤 2...
```

## 根本原因

### 1. 模型的"安全行为"模式

模型在每次 thinking 结束时都会执行：
- 更新 TodoWrite（确认进度）
- Read 调研结果（确认状态）
- 重新制定方案（确认计划）

这种"确认-再确认"的行为模式导致了无限循环。

### 2. 缺乏"已完成"状态感知

模型没有正确识别：
- 调研结果已经返回并记录到文件
- TodoWrite 已经更新过（且没有持久化）
- Read 结果没有变化（`Wasted call` 提示被忽略）

### 3. 等待 Agent 调用的重复检查

在等待 command-line-expert 和 python-backend-expert 的调研结果时，模型可能：
- 不断检查调用状态
- 每次检查都触发 TodoWrite 和 Read
- 形成"检查-更新-再检查"的循环

## 触发条件

- Manager agent 处理多步骤任务
- 任务需要等待其他 Agent 的调用结果
- 任务涉及文件读取和进度更新
- 上下文较长，模型丢失了"已完成"的判断

## 影响范围

- **上下文空间**: 浪费约 150K-180K tokens（约 80-90% 上下文）
- **功能正确性**: 不影响，但导致任务无法在当前会话完成
- **用户体验**: 需要在新会话中重新开始

## 解决方案

### 短期方案（模型行为层面）

1. **限制 TodoWrite 调用频率**: 同一任务状态更新间隔不少于 5 分钟
2. **检测重复 Read**: 如果文件未变化，跳过后续处理
3. **识别"已完成"状态**: 调研结果返回后，直接进入实现阶段

### 长期方案（系统架构层面）

1. **TodoWrite 持久化**: 将 TodoWrite 任务持久化到 TaskList，避免重复创建
2. **Heartbeat 机制**: 在 Agent 循环卡住时触发 heartbeat 提醒
3. **上下文压缩**: 自动检测并移除重复的工具调用记录

3. 检查是否有连续相同的 thinking 内容

## 经验教训

1. **模型需要"停止条件"**: 无限循环的根本原因是模型缺乏明确的停止条件
2. **工具调用应该有幂等性检查**: 如果状态没有变化，不应该重复执行
3. **上下文管理很重要**: 长对话中，模型容易丢失"已完成"的判断
4. **TodoWrite 应该持久化**: 临时的 TodoWrite 在会话重启后会丢失
5. **需要 heartbeat 机制**: 在 Agent 卡住时提供外部干预

## 建议的后续行动

1. 在 `docs/coding-rules/` 中添加"避免工具调用循环"的规则
2. 考虑在 Agent Bridge 中添加工具调用频率限制
3. 研究如何让模型更好地识别"已完成"状态
4. 在新会话中继续 OpenCode 集成任务
