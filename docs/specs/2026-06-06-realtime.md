## Overview

Realtime 模块提供进程级 WebSocket 连接管理能力，以房间（room）为单位对连接进行分组，支持向指定群聊房间广播实时信号。

## Scope

**职责边界**：

- WebSocketManager：管理连接的加入、移除和按房间广播
- 房间模型：按 `group_chat_id` 隔离连接，自动清理空房间
- 依赖注入：进程级单例 + FastAPI `Depends` 注入
- 事件模型：定义 RefreshSignal 数据结构
- 异常体系：WebSocket 相关异常的继承层次

**不负责**：

- WebSocket 端点的路由和生命周期管理（由 `api/websocket` 模块负责）
- 群聊业务逻辑（由 core/orchestration 层负责）
- 消息持久化

## Core Behavior

### 连接生命周期

**connect**：接受 WebSocket 连接，将其加入指定 `group_chat_id` 对应的房间列表。如果房间不存在则自动创建。

**disconnect**：将连接从房间中移除。如果移除后房间为空，删除该房间以释放资源。对不存在的房间或连接做静默忽略。

### 房间模型

- 房间以 `group_chat_id` 为键，值为该房间内的 WebSocket 连接列表
- 一个房间可以有零到多个连接（同一群聊可能有多个客户端）
- 房间在首个连接加入时创建，在最后一个连接离开时自动销毁

### 广播机制

向指定房间的所有连接发送 JSON 消息。广播过程中：

1. 遍历房间内所有连接逐个发送
2. 发送失败的连接被收集到失败列表
3. 广播完成后，将失败连接从房间中移除
4. 如果清理后房间为空，删除房间
5. 向空房间广播时记录警告日志并跳过

### 依赖注入模式

- WebSocketManager 以进程级单例存在，通过 `get_realtime_manager()` 获取
- 单例在首次调用时惰性初始化
- 提供 `reset_realtime_manager()` 用于测试环境重置
- API 端点通过 FastAPI 的 `Depends(get_realtime_manager)` 注入管理器实例
- `broadcast_group_chat_refresh()` 是便捷函数，内部构造 RefreshSignal 并调用广播

## Technical Contract

### WebSocketManager 接口

| 方法 | 语义 |
|------|------|
| `connect` | 接受 WebSocket 连接并加入指定房间 |
| `disconnect` | 从房间移除连接，空房间自动清理 |
| `broadcast` | 向房间内所有连接广播 JSON 消息，失败连接自动清理 |

### RefreshSignal schema

| 字段 | 类型 | 说明 |
|------|------|------|
| `type` | str | 信号类型，固定为 `"refresh"` |
| `group_chat_id` | str | 触发刷新的群聊 ID |
| `timestamp` | datetime | 信号生成时间戳，自动生成 |

`make_refresh_signal()` 工厂函数接受 `group_chat_id`，返回 RefreshSignal 实例。

### 异常体系

异常采用双重继承设计，同时继承 WebSocket 基类和对应的通用异常基类：

```
AgentsHubError
├── WebSocketError                          # WebSocket 异常基类
│   ├── WebSocketConnectionError            # 连接错误（+ ExternalServiceError）
│   ├── WebSocketRoomNotFoundError          # 房间不存在（+ ResourceNotFoundError）
│   ├── WebSocketBroadcastError             # 广播错误（+ ExternalServiceError）
│   └── WebSocketValidationError            # 校验错误（+ ValidationError）
```

每个 WebSocket 异常同时继承 `AgentsHubError` 子树和 `WebSocketError`，使得上层可以按通用异常类型或 WebSocket 异常类型统一捕获。
