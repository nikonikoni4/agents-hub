# Teams 团队管理模块

## 版本

| 版本 | 更新内容 |
| ---- | -------- |
| 1.0 | 从设计稿重写为正式 spec 初稿 |

## Overview

teams 模块是 agents-hub 系统的**团队配置管理层**，负责团队的全生命周期管理，包括创建、查询、更新和删除。

模块定位：
- **负责**：团队 CRUD、成员列表管理、成员验证（调用 RoleManager）、持久化到 JSON 文件、通过 HTTP API 暴露功能
- **不负责**：团队的运行时调度逻辑、消息传递、群聊管理、角色配置管理

核心设计原则：
- **SSOT**：`teams.json` 是团队数据的唯一来源
- **简单性**：团队只是角色名称的集合，用于创建群聊时作为预设
- **验证优先**：添加成员时必须验证角色是否存在

## Scope

### 范围内

- 团队的创建、删除、查询、列表、更新
- 团队成员列表管理（role 名称列表）
- 成员验证（调用 RoleManager 验证 role 是否存在）
- 持久化到 `config.data_path/teams/teams.json`
- 并发安全的文件读写
- HTTP API 接口

### 范围外

- 团队的运行时调度逻辑
- 团队与群聊的绑定关系（由 GroupChat 负责）
- 角色配置的 CRUD（由 roles 模块负责）
- 消息传递和会话管理
- 团队权限管理

## Core Behavior

### 团队的本质

团队是**角色名称的集合**，在创建群聊时作为预设，将 `members` 列表传入 GroupChat 的初始化参数。

### 数据模型

**TeamInfo**（领域模型）：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | str | 是 | 团队名称（唯一标识） |
| members | list[str] | 是 | 成员角色名称列表，至少包含一个成员 |

### 持久化格式

保存到 `config.data_path/teams/teams.json`，采用**数组格式**：

```json
[
  {
    "name": "frontend-team",
    "members": ["alice", "bob"]
  },
  {
    "name": "backend-team",
    "members": ["charlie", "david"]
  }
]
```

### 成员验证机制

**验证时机**：
- 创建团队时验证所有成员
- 更新团队成员列表时验证新成员列表

**验证规则**：
- 调用 RoleManager 获取所有可用角色，检查成员列表中的每个角色是否存在
- 成员列表为空时拒绝操作
- 成员中包含不存在的角色时拒绝操作，并返回无效成员列表和可用角色列表

### 名称唯一性

团队名称作为唯一标识，不允许重复：
- 创建时检查名称是否已存在，存在则拒绝
- 更新名称时检查新名称是否与其他团队冲突

### 文件初始化

- 如果 `teams/` 目录不存在，自动创建
- 如果 `teams.json` 不存在，自动创建并写入空数组 `[]`

### 并发控制

使用锁机制保护 `teams.json` 文件的读写操作，确保读-修改-写的原子性。

## Technical Contract

### Request Schemas

**TeamCreateRequest**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | str | 是 | 团队名称 |
| members | list[str] | 是 | 成员角色名称列表 |

**TeamUpdateRequest**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | str \| None | 否 | 新的团队名称，为 None 时保持原名称 |
| members | list[str] \| None | 否 | 新的成员列表，为 None 时保持原成员列表 |

所有字段可选，仅更新传入的字段（PATCH 语义）。

### Response Schema

**TeamResponse**

| 字段 | 类型 | 说明 |
|------|------|------|
| name | str | 团队名称 |
| members | list[str] | 成员角色名称列表 |

### 异常类型

| 异常 | 基类 | 触发场景 | HTTP 状态码 |
|------|------|----------|------------|
| TeamNotFoundError | ResourceNotFoundError | 获取/更新/删除不存在的团队 | 404 |
| TeamAlreadyExistsError | ValidationError | 创建时名称已存在，或更新时新名称冲突 | 409 |
| InvalidTeamMembersError | ValidationError | 成员列表包含不存在的角色 | 422 |
| EmptyTeamMembersError | ValidationError | 成员列表为空 | 422 |

所有异常继承自项目统一异常基类，由全局异常处理器转换为 HTTP 错误响应。

### API 端点

所有端点挂在 `/teams` 前缀下。

| 方法 | 路径 | 说明 | 成功状态码 |
|------|------|------|-----------|
| GET | `/teams` | 列出所有团队 | 200 |
| GET | `/teams/{name}` | 获取单个团队 | 200 |
| POST | `/teams` | 创建团队 | 201 |
| PATCH | `/teams/{name}` | 更新团队信息 | 200 |
| DELETE | `/teams/{name}` | 删除团队 | 200 |

### API 架构分层

```
route -> service -> manager
```

- **route**：HTTP 入口，只做参数接收和响应转换
- **service**：业务协调，编排 manager 调用
- **manager**：领域逻辑

路由通过 FastAPI `Depends` 注入 Service 实例，禁止在路由中直接实例化。路由层禁止 try/except，所有异常由全局异常处理器统一处理。

## Acceptance Notes

1. 能创建团队，自动创建 `teams/` 目录和 `teams.json` 文件
2. 能列出所有团队，返回正确的 TeamResponse 列表
3. 能按名称获取单个团队
4. 能更新团队名称和成员列表（PATCH 语义，仅更新传入字段）
5. 能删除指定团队
6. 创建/更新时验证成员是否存在，不存在则返回 422
7. 成员列表为空时返回 422
8. 创建/更新时检查名称冲突，冲突时返回 409
9. 获取/更新/删除不存在的团队时返回 404
10. `teams.json` 格式正确（数组形式），持久化到正确路径
11. 并发读写时数据不丢失
12. 异常正确映射为 HTTP 错误响应
13. Route 层不包含业务逻辑和 try/except

## Out of Spec

以下内容不在本 spec 中长期维护：

1. 团队的运行时调度逻辑（由 core/orchestration 负责）
2. 团队与群聊的绑定关系管理
3. 角色配置的 CRUD 操作（由 roles 模块负责）
4. 团队权限管理
5. 团队模板功能
6. 团队成员的角色分配（manager/worker 等）
