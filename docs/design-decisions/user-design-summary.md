# User Design Summary

## 目的

记录用户在重大设计决策中的偏好、判断方式和风险倾向，为后续 agent 自主决策提供参考，也为用户复盘决策方式提供材料。

## 决策偏好总结

### 已观察到的稳定偏好

1. **调研驱动决策**：不会凭直觉选择技术方案，会做对比调研后再决策。即使初始调研有遗漏（未发现 SDK），也会在发现后重新评估，而不是坚持已有方案。

2. **"够用就好"原则**：当模块定位为接口层/执行层时，倾向于选择最简单的集成方式。不为"可能需要"的能力引入额外复杂度和依赖。

3. **依赖控制意识**：对引入新依赖（尤其是实验性依赖）持谨慎态度。即使 SDK 能力更强，也优先考虑零新依赖的方案。

4. **一致性偏好**：倾向于让同类模块保持一致的接入方式（Claude 和 Codex 都用 CLI subprocess），降低认知和维护成本。

5. **安全性和职责边界优先**：在架构设计中，安全性（避免越权）和职责边界清晰度的优先级高于实现简单性。愿意为此新增代码（如 MessageBus）。当用户明确提出"原则"时，这些原则是硬约束，不是优化目标。

6. **批判性参考开源框架**：不会因为"知名框架这样做"就接受方案，而是会深入分析其问题。对 MetaGPT 和 AutoGen 的方案都提出了明确的反对理由，而非盲目采用。

7. **控制面优先于展示层**：在多 Agent 协作中，用户倾向把指令、调用、状态机作为事实来源，把群聊历史视为公开协作记录或投影。展示层记录不应反向驱动核心调用生命周期。

8. **语义拆分优先于工具复用**：当同一个工具会同时表达"说话"和"完成任务"时，用户倾向拆分为不同工具，避免一个参数或上下文隐式改变工具含义。

9. **入口与共享能力分离**：当同一能力被 API 和 MCP 等多个入口复用时，用户倾向抽出独立模块，而不是让一个入口模块成为另一个入口的运行时依赖。

10. **运行期系统采用内存 SSOT**：对于长生命周期的运行态状态（session、context load state 等），用户认可"内存是运行期 SSOT、文件是持久化副本"的成熟模式，愿意接受新增抽象层（Runtime/State）的复杂度来换取性能和清晰的职责边界。但配置/资源类数据（roles/skills）仍保持文件优先。

11. **显式错误暴露而非静默继续**：在持久化失败场景中，用户选择"设置 degraded 标记 + 抛出异常"而非"尝试回滚或静默继续"，体现出对 Agent 副作用不可回滚的现实认知和务实态度。

12. **状态分类策略**：用户不追求统一的 SSOT 策略，而是根据数据特点（运行态/配置态/历史索引）选择不同策略，体现了"合适优于一致"的工程务实性。

13. **职责分离 + 声明式配置**：当函数内部硬编码大段角色特定内容时，用户倾向提取到子类作为类变量，让函数只做编排。适用于提示词管理、配置管理等需要按角色差异化的场景。

14. **从系统实际能力出发设计约束**：不会从"理想行为"出发设计工具语义，而是先确认系统有没有对应的能力（如中断机制）。当发现某工具在当前架构下没有实际作用时，会果断收窄其语义。

15. **约束是系统工程需要，不是对 agent 能力的不信任**：agent 本身能力很强，但不受约束会导致系统熵增严重。约束的对象是接口和执行路径，不是执行本身。

### 偏好成立的上下文条件

- 项目处于 MVP / 早期阶段
- 模块职责边界已明确（如 agent_bridge 定位为纯执行层）
- 当前方案能覆盖已知需求
- 候选方案引入的是实验性或未充分验证的依赖

### 可能导致误判的倾向

- **低估"纯接口层"向"控制面"演进的速度**：当产品开始需要多轮交互、中断、转向等能力时，CLI exec 模型的局限会快速暴露。如果触发条件判断滞后，可能在需要迁移时才发现 SDK 也有 breaking changes。
- **调研遗漏的系统性风险**：本次决策的起点是"初始调研未发现 SDK"。这提示调研流程可能需要更系统化（如检查 PyPI、搜索 GitHub org、查看上游仓库的 sdk/ 目录），而不是依赖单次搜索结果。
- **对"耦合"概念的过度敏感**：在消息架构讨论中，用户最初认为"Agent 持有 GroupChatContext 引用"是耦合，但实际上这是标准的依赖注入。这种过度敏感可能导致设计过度复杂化。需要区分"依赖注入"（合理）和"双向引用 + 直接调用方法"（真正的耦合）。
- **追求"完美设计"的风险**：用户在架构设计中对安全性和职责边界的要求很高，这是优点，但也可能导致过度设计。需要在"原则性问题"（如越权访问）和"实现细节"（如是否需要中心化队列）之间找到平衡。
- **工具数量增长的选择成本**：为了语义清晰而拆分工具是合理的，但每新增一个工具都会提高 LLM 选择成本。需要用清晰命名、互斥前提和失败提醒机制控制复杂度。
- **提前抽象的边界**：用户对模块边界和循环依赖风险敏感，这通常能避免后续架构缠绕；但如果没有真实复用入口或依赖风险，过早抽象也可能增加文件和概念数量。

## 给后续 agent 的参考

### 可以参考的判断原则

- 当模块定位为"纯执行层/接口层"时，优先选择最简单的集成方式
- 对实验性依赖保持谨慎，除非当前方案已无法覆盖需求
- 保持同类模块的接入方式一致性
- **在架构设计中，安全性和职责边界优先于实现简单性**——这是用户的核心价值观
- **批判性参考开源框架**——不盲目采用知名框架的方案，而是分析其适用性和问题
- **控制面事实优先**——AgentCall、任务状态、call_id 等控制面对象应作为生命周期事实来源；群聊消息主要承担可见记录和协作上下文职责
- **避免语义过载工具**——如果一个工具既像"发言"又像"回执"，应优先拆开或明确状态机边界
- **共享运行时能力不应藏在单一入口模块下**——当 API 和 MCP 都要使用同一实时通知能力时，应优先抽出独立边界，而不是让 MCP 依赖 API
- **对于运行期系统，优先采用内存 SSOT + 同步持久化**——这是成熟的设计模式，性能和职责边界都更优。但需要区分运行态、配置态和历史索引，不同类型采用不同策略
- **持久化失败必须显式暴露**——通过 error flag + 异常的方式让失败可见，避免系统在"内存已前进、文件没跟上"的状态下静默运行
- **务实地接受"不完美的一致性"**——在 Agent 副作用不可回滚的前提下，不追求完整的事务回滚，而是通过守卫检查阻止 degraded 状态下的高风险操作

### 必须重新确认的前提

- "只当接口使用"的定位是否仍然成立——如果 agents-hub 开始需要 interrupt/steer/thread 生命周期管理，需要立即重新评估
- SDK 是否已从实验性变为稳定版——如果 SDK 稳定，迁移成本评估会变化
- CLI JSONL parser 的维护成本是否在可控范围内
- **用户提出的"原则"是否仍然适用**——如"避免越权访问"在当前场景下是否仍是硬约束
- **工具数量是否仍然可控**——如果 MCP 工具继续增加，需要重新确认"语义拆分"带来的清晰度是否仍大于 tool selection 成本
- **抽出的共享模块是否有真实复用者**——如果只有 API 一个入口使用，就不必为了抽象而抽象；如果 MCP、API 或其他入口都需要使用，则独立边界更合理
- **内存 SSOT 的持久化失败率是否可控**——如果文件系统频繁故障或保存失败率高，需要重新评估同步持久化策略的可行性
- **运行态状态的分类边界是否清晰**——在扩展新功能时，必须明确哪些属于运行态（内存 SSOT）、哪些属于配置态（文件优先），避免混淆导致错误的 SSOT 策略

### 不建议直接沿用的做法

- 不建议在所有场景下都遵循"最简单方案"原则——对于长期运行的服务、高频调用场景、需要精细控制的场景，SDK 可能是更合理的选择
- 不建议仅因为"和 Claude 侧一致"就拒绝 Codex 侧的差异化接入——两个平台的 SDK 成熟度和能力模型不同，可以接受不对称的后端策略
- **不建议将"依赖注入"误判为"耦合"**——Agent 持有共享资源（如 GroupChatContext）的引用是合理的依赖注入，只要 Agent 不修改其他 Agent 的状态即可
- **不建议为了"完美设计"而过度复杂化**——在满足核心原则（如避免越权）的前提下，优先选择简单方案。例如，是否需要中心化暂存队列应该基于实际需求，而非理论上的"解耦"
- **不建议把所有跨模块协作都升级成事件总线**——独立 realtime 边界已经能解决当前 API/MCP 依赖问题；只有当事件类型和订阅者明显增加时，再考虑更完整的 event bus
- **不建议在所有状态上统一 SSOT 策略**——运行态适合内存 SSOT，配置态适合文件优先，历史索引可以用 read model。根据数据特点选择策略，而非追求架构一致性
- **不建议为了"完美一致性"引入过重的持久化方案**——Event Sourcing、WAL、分布式事务等方案在 Agent 副作用不可回滚的前提下收益有限，当前阶段的同步持久化 + 显式错误暴露已经足够务实
