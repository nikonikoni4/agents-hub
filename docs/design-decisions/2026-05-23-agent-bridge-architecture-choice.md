# Agent Bridge 架构设计决策

| 版本 | 更新内容 |
| ---- | -------- |
| 1.0 | 创建文档初稿，确定采用扁平化架构（方案B） |

## 一、问题界定

### 问题简述

agents-hub 需要设计一个统一的 agent_bridge 模块，用于调用不同 AI 平台的 CLI 工具（Claude Code、Codex）。核心问题是：**如何设计模块架构，使其既能满足当前需求，又具备良好的扩展性和可维护性？**

### 讨论范围

**范围内**：
- agent_bridge 模块的整体架构设计
- 职责划分方式（继承 vs 组合）
- 代码组织结构（按平台分文件 vs 按职责分目录）
- 扩展性和可维护性的权衡
- 设计模式选择（抽象类继承 vs Protocol 组合）

**范围外**：
- 具体的错误处理实现细节（留白，之后实现）
- 性能优化策略
- 非流式输出模式（当前只采用流式输出）
- 配置动态变更机制（配置固定，作为初始化参数传入）
- 会话管理的具体实现（session_id 由调用方传入）

### 关键信息明确定义

| 关键概念 | 明确定义 |
|---------|---------|
| **配置管理** | 角色配置（system prompt、skill 选择）固定，作为初始化参数传入 |
| **输出模式** | 只采用流式输出，不考虑非流式场景 |
| **模块定位** | 纯执行层，负责启动 CLI 和解析输出，不涉及业务逻辑 |
| **会话管理** | session_id 由调用方传入，模块本身不管理会话生命周期 |

### 问题深度

这是一个**架构设计层面的深度讨论**，涉及：
- 设计模式选择（继承 vs 组合）
- 架构原则应用（SRP、DRY、组合优于继承）
- 长期可维护性和扩展性权衡
- 代码组织哲学（按平台 vs 按职责）

## 二、现状（当前的问题）

### 2.1 背景

agents-hub 需要支持多个 AI 平台的 CLI 调用，当前面临以下挑战：

1. **平台差异性**：不同平台的 CLI 命令格式、输出格式、配置方式各不相同
2. **统一接口需求**：上层调用方需要一个统一的接口，屏蔽平台差异
3. **未来扩展性**：可能需要支持更多平台（如 Gemini、OpenAI 等）
4. **职责边界模糊**：执行和解析逻辑是否应该耦合在一起？

### 2.2 核心矛盾

- **简单性 vs 扩展性**：简单的继承架构易于理解，但扩展时可能导致职责混乱
- **内聚性 vs 灵活性**：将执行和解析放在一个类中内聚性强，但灵活性差
- **代码组织**：按平台分文件 vs 按职责分目录，哪种更利于维护？

## 三、可选方案

### 方案 A：继承架构

#### 设计思路

**核心理念**：一个 Agent 类 = 完整的执行单元（包含执行 + 解析）

#### 目录结构

```
backend/agent_bridge/
├── config.py          # RoleConfig（配置数据类）
├── base.py            # BaseAgent（抽象基类）
├── claude.py          # ClaudeAgent（继承 BaseAgent）
├── codex.py           # CodexAgent（继承 BaseAgent）
└── bridge.py          # AgentBridge（统一封装）
```

#### 职责划分

| 模块 | 职责 |
|------|------|
| **BaseAgent** | 定义接口契约（抽象方法） |
| **ClaudeAgent** | 实现完整流程：构建命令 + 启动进程 + 解析输出 |
| **CodexAgent** | 实现完整流程：构建命令 + 启动进程 + 解析输出 |
| **AgentBridge** | 根据 platform 选择对应的 Agent，代理调用 |

#### 优势

1. **代码路径短**：一个类搞定所有事，调用链简单
2. **内聚性强**：执行和解析逻辑在同一个类中，相关代码集中
3. **易于理解**：符合传统 OOP 思维，新手容易上手
4. **按平台组织**：每个平台一个文件，查找代码直观

#### 劣势

1. **职责混杂**：违反 SRP 原则，一个类既负责执行又负责解析
2. **扩展困难**：
   - 如果只需修改命令构建逻辑，必须修改整个 Agent 类
   - 如果多个平台输出格式相同，无法复用解析器
3. **测试复杂**：测试时必须同时处理进程启动和输出解析
4. **继承层次风险**：未来如果需要共享部分逻辑，可能导致类层次复杂化

---

### 方案 B：扁平化架构（推荐）

#### 设计思路

**核心理念**：执行和解析分离，通过组合而非继承实现复用

#### 目录结构

```
backend/agent_bridge/
├── config.py          # RoleConfig
├── protocols.py       # Executor, Parser 协议定义
├── executors/
│   ├── claude.py      # ClaudeExecutor
│   └── codex.py       # CodexExecutor
├── parsers/
│   ├── claude.py      # ClaudeParser
│   └── codex.py       # CodexParser
└── bridge.py          # AgentBridge
```

#### 职责划分

| 模块 | 职责 |
|------|------|
| **Executor** | 构建命令 + 启动进程 + 返回原始输出流 |
| **Parser** | 解析原始输出 + 转换为统一格式 |
| **AgentBridge** | 根据 platform 选择 Executor 和 Parser，组装流程 |

#### 优势

1. **职责清晰**：符合 SRP 原则，Executor 只管执行，Parser 只管解析
2. **扩展性强**：
   - 修改命令构建逻辑 → 只改 Executor
   - 修改输出解析逻辑 → 只改 Parser
   - 多个平台输出格式相同 → 直接共享 Parser
3. **测试简单**：可以独立测试 Executor（mock 进程输出）和 Parser（mock 原始数据）
4. **组合优于继承**：通过组合实现功能，灵活性更高
5. **代码组织直观**：按职责分目录，一看就知道每个模块的作用

#### 劣势

1. **代码路径稍长**：需要组合 Executor + Parser，调用链多一层
2. **文件数量增加**：每个平台需要两个文件（Executor + Parser）

## 四、最终决策

### 决策结论

**选择方案 B：扁平化架构**

### 决策原因

#### 1. 符合项目核心编码原则

根据 `CLAUDE.md` 中的核心编码原则：

- **SRP（单一职责原则）**：方案 B 将执行和解析分离，每个模块只做一件事
- **DRY（不要重复自己）**：通过组合实现复用，避免重复逻辑
- **组合优于继承**：方案 B 使用组合，灵活性更高

方案 A 违反了 SRP 原则，一个 Agent 类承担了两个职责。

#### 2. 扩展性更强

当前需求虽然简单（只有流式输出、配置固定），但未来可能面临：

- **新增平台**：如 Gemini、OpenAI 等
- **输出格式变化**：某个平台升级后输出格式改变
- **解析器复用**：多个平台可能使用相同的输出格式

方案 B 在这些场景下只需修改局部模块，而方案 A 需要修改整个 Agent 类。

#### 3. 可测试性更好

- **方案 B**：可以独立测试 Executor 和 Parser，测试用例更简单
- **方案 A**：测试时必须同时处理进程启动和输出解析，测试复杂度更高

#### 4. 代码组织更清晰

```
executors/  # 一看就知道是启动 CLI 的
parsers/    # 一看就知道是解析输出的
```

比方案 A 的 `claude.py`、`codex.py` 更容易理解每个文件的职责。

#### 5. 权衡分析

方案 A 的唯一优势是"代码路径短"，但这在当前需求下（只有流式输出、配置固定）并不是关键优势。相反，方案 B 在可维护性、可测试性、扩展性上的优势更符合项目长期发展需求。

---

## 五、实施建议

### 5.1 实施步骤

1. **定义 Protocol**：在 `protocols.py` 中定义 `Executor` 和 `Parser` 接口
2. **实现 Claude 支持**：先实现 `ClaudeExecutor` 和 `ClaudeParser`
3. **实现 Codex 支持**：实现 `CodexExecutor` 和 `CodexParser`
4. **实现 AgentBridge**：组装 Executor 和 Parser，提供统一接口
5. **编写测试**：分别测试 Executor 和 Parser

### 5.2 注意事项

- 使用 `Protocol` 而非抽象类定义接口，保持 Python 风格
- Executor 返回原始输出流，不做任何解析
- Parser 只负责解析，不涉及进程管理
- AgentBridge 负责组装流程，但不包含业务逻辑

### 5.3 未来扩展方向

- 如果多个平台输出格式相同，可以共享 Parser
- 如果需要支持非流式输出，可以在 Executor 中添加新方法
- 如果需要动态配置，可以在 AgentBridge 中添加配置更新接口
