# Agent Hook 适用边界

## 版本

| 版本 | 更新内容 |
| ---- | -------- |
| 1.0 | 创建文档初稿 |

## 问题界定

### 问题简述

用户原本计划通过 Hook 机制实现两个功能：(1) Before Hook 根据指令检索相关上下文注入给 Agent，减少 token 消耗；(2) After Hook 完成后进行一轮审查。这两个功能都涉及大模型调用，需要明确 hook 机制是否适合承载这类重型功能。

### 讨论范围

- Agent Hook 机制的适用场景和边界
- 重型功能（大模型调用）应该放在 hook 还是显式工作流
- hook 的设计模式选择

### 非讨论范围

- hook 的具体实现代码
- 其他 Agent 功能的设计

### 问题深度

这是架构原则层面的决策——hook 的定位和职责边界，会影响后续所有 hook 相关的设计。

## 现状

- Agent Hook 机制尚未实现，处于设计阶段
- 用户有两个具体需求：上下文自动注入、结果自动审查
- 这两个需求都涉及大模型调用，执行时间长、结果不可预测

## 可选方案

### 方案 A：Hook 承载重型功能

将上下文检索、结果审查等功能实现为 before/after hook，在 Agent 调用流程中自动触发。

**优势**

- 功能自动触发，调用方无需显式调用
- 统一的 hook 接口，所有扩展点一致

**劣势**

- 大模型调用放在 hook 里不可控（执行时间、结果质量）
- hook 失败时难以处理（重试？跳过？中断？）
- 调用方无法感知 hook 内部发生了什么
- 维护成本高，调试困难

### 方案 B：Hook 只做轻量检查，重型功能放显式工作流

hook 只负责简单的输入模型检查、输入输出约束；重型功能（大模型调用、上下文检索）在交互工作流中显式调用。

**优势**

- 职责清晰：hook 做检查，工作流做处理
- 重型功能显式调用，可控可调试
- hook 简单稳定，不易出错

**劣势**

- 调用方需要显式调用重型功能
- 可能有代码重复（如果多个地方需要相同的审查逻辑）

## 最终决策

选择 **方案 B：Hook 只做轻量检查，重型功能放显式工作流**。

暂缓 hook 机制的实现，等后续有明确的轻量级 hook 需求时再实现。

## 决策原因

- **可控性**：大模型调用放在 hook 里不可控，执行时间长、结果不可预测，失败时难以处理
- **可维护性**：hook 应该是简单稳定的，重型功能放在 hook 里增加维护成本
- **可调试性**：显式调用比隐式触发更容易调试和追踪
- **符合 hook 本质**：hook 的传统定位是轻量级的拦截和检查，不是复杂业务逻辑

## 后续影响

- 暂不实现 hook 机制
- 上下文检索、结果审查等功能如果需要，直接在 Agent 调用流程中显式实现
- 后续如果有轻量级 hook 需求（如输入格式校验、权限检查），可以重新评估
