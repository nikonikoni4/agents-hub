3. Memory 机制详情

3.1 写入机制

关键发现：标准 Memory 接口没有自动写入机制！
AssistantAgent 的处理流程只有读取（STEP 2: memory.update_context()），没有自动调用 memory.add()。写入必须由外部显式调用或通过 Task-Centric Memory 的学习机制触发。

写入时机与方式

方式时机触发方
手动预加载agent 初始化前外部代码显式调用 memory.add()
从失败中学习任务执行失败后Task-Centric Memory 的 train_on_task()
从用户消息提取处理用户消息时Task-Centric Memory 的 consider_memo_storage()

Task-Centric Memory 写入流程

flowchart TD Input[用户消息/任务失败] --> ExtractTask[提取任务 extract_task prompt] ExtractTask --> IsFailure{是否失败?} IsFailure -- 是 --> LearnFromFailure[从失败中学习] IsFailure -- 否 --> ExtractAdvice[提取建议] LearnFromFailure --> Generalize[泛化任务] ExtractAdvice --> Generalize Generalize --> FindTopics[提取索引主题] FindTopics --> Store[存储 洞察+主题+任务] Store --> Persist[持久化到磁盘]

3.2 读取机制

读取是自动的！
在 AssistantAgent 的 STEP 2，每次推理前都会自动调用 memory.update_context(model_context)，将相关记忆注入上下文。

不同实现的读取策略

实现查询方式注入方式
ListMemory全量返回（按时间顺序）格式化为 SystemMessage 注入
ChromaDB/Redis/Mem0用最后一条消息做语义相似度查询按分数和阈值过滤后注入
Task-Centric Memory泛化任务 → 提取主题 → 检索 → LLM 验证验证通过的记忆注入

4. 工具调用机制详情

4.1 Tool Schema 格式

class ToolSchema(TypedDict):
    name: str                                    # 工具名称
    description: NotRequired[str]                # 工具描述
    parameters: NotRequired[ParametersSchema]    # 参数定义
    strict: NotRequired[bool]                    # 严格模式

class ParametersSchema(TypedDict):
    type: str                                    # 固定为 "object"
    properties: Dict[str, Any]                   # 参数属性定义
    required: NotRequired[Sequence[str]]         # 必填参数列表
    additionalProperties: NotRequired[bool]      # 是否允许额外参数

实际 Schema JSON 示例

// Python 函数
async def get_stock_price(ticker: str, date: Annotated[str, "Date in YYYY/MM/DD"]) -> float:
    return random.uniform(10, 200)

// 生成的 ToolSchema
{
    "name": "get_stock_price",
    "description": "Fetch the stock price for a given ticker.",
    "parameters": {
        "type": "object",
        "properties": {
            "ticker": { "type": "string", "description": "ticker" },
            "date": { "type": "string", "description": "Date in YYYY/MM/DD" }
        },
        "required": ["ticker", "date"],
        "additionalProperties": false
    },
    "strict": false
}

4.3 工具错误处理机制（三层）

层次错误类型处理方式结果
Workbench 层工具不存在遍历所有工具未命中ToolResult(is_error=True, "Tool {name} not found.")
Workbench 层执行异常try-except 捕获ToolResult(is_error=True, "{exception_message}")

工具调用数据流

用户消息
  -> AssistantAgent.on_messages_stream()
  -> _call_llm() 传递 tools=[tool.schema for tool in workbench]
  -> LLM 返回 FunctionCall(id, name, arguments)
  -> _execute_tool_call()
     -> json.loads(arguments) 解析参数

        -> tool.run_json(args, cancellation_token)
           -> args_type.model_validate(args) Pydantic 验证

     -> FunctionExecutionResult(content, call_id, is_error, name)
  -> model_context.add_message(FunctionExecutionResultMessage)
  -> 下一轮 LLM 推理或总结

5. Agent 间通信机制详情

5.1 Manager → Worker 消息格式

关键发现：Manager 不直接"发送"任务内容给 Worker！
Manager 仅发送 GroupChatRequestPublish（空消息）作为触发信号。Worker 通过自主缓存的 _message_buffer 获取完整上下文。

核心消息类型

class GroupChatStart(BaseModel):
    """启动群聊的请求"""
    messages: List[BaseChatMessage] | None = None  # 初始任务消息
    output_task_messages: bool = True

class GroupChatRequestPublish(BaseModel):
    """请求发布消息到群聊"""
    ...  # 空模型，仅作为触发信号

class GroupChatAgentResponse(BaseModel):
    """Agent 的响应"""
    response: Response  # 包含 chat_message 和 inner_messages
    name: str           # 产生响应的 Agent 名称

6. 四种发言模式与 Buffer 机制

6.1 四种发言模式

Manager 始终是"点名"的人，但"点名规则"由 Team 类型决定。

Team 类型顺序决定方式确定性类比
RoundRobinGroupChat按参与者列表顺序轮流高（固定顺序）圆桌会议，按座位顺序发言
SelectorGroupChatLLM 或自定义函数选择中（依赖 LLM）有主持人的研讨会，根据讨论内容决定
Swarm通过 HandoffMessage 决定中（依赖 Agent 决策）接力赛，当前跑者决定交给谁
MagenticOneGroupChatOrchestrator 规划中（依赖规划能力）项目经理，规划任务分配

RoundRobinGroupChat：固定轮转

class RoundRobinGroupChatManager(BaseGroupChatManager):
    def __init__(self, ...):
        self._current_index = 0

    async def select_speaker(self, thread):
        speaker = self._participant_names[self._current_index]
        self._current_index = (self._current_index + 1) % len(self._participant_names)
        return [speaker]

# 创建时指定顺序
team = RoundRobinGroupChat(

    ...
)

async def select_speaker(self, thread):
    # 方式1: 自定义函数优先

Buffer 更新机制

class ChatAgentContainer:
    def __init__(self, ...):
        self._message_buffer: List[BaseAgentEvent | BaseChatMessage] = []

    @event  # 订阅 group_topic
    async def handle_start(self, message: GroupChatStart, ctx: MessageContext):
        """收到 GroupChatStart，存入 buffer"""
        if message.messages:
            self._message_buffer.extend(message.messages)

    @event  # 订阅 group_topic
    async def handle_agent_response(self, message: GroupChatAgentResponse, ctx: MessageContext):
        """收到其他 Agent 的响应，也存入 buffer"""
        self._message_buffer.append(message.response.chat_message)

Buffer 内容：群聊聊天记录

Buffer 只记录最终发言，不包含中间过程
Agent 的中间工具调用和思考过程不会写入 buffer，只有最终结果发布到 group_topic。 
# Buffer 内容示例
message_buffer = [

    TextMessage(content="我查到这些酒店...", source="Hotel_Agent"),           # HA 发言
    TextMessage(content="我查到这些航班...", source="Flight_Agent"),          # FA 发言
]

为什么 Buffer 要独立？

原因说明
组件化设计每个 Container 是独立的组件，可以单独测试、替换
状态隔离每个 Container 有自己的状态（buffer、Agent 实例）
并发安全多个 Container 可以并发处理消息，互不干扰
可扩展性可以动态添加/移除 Container

设计哲学

AutoGen 的核心假设：多 Agent 协作应该像人类团队开会，而不是经理打电话派活。

AutoGen：像一个开放办公室，所有人坐在一起，谁说话所有人都能听到。
DeerFlow/CrewAI：像一个经理办公室，经理一个个叫人进来谈话，每个人只知道经理告诉他的内容。

Topic 和 Subscription 机制

@dataclass(eq=True, frozen=True)
class TopicId:
    type: str     # 事件类型
    source: str   # 事件来源上下文

class TypeSubscription(Subscription):
    def is_match(self, topic_id: TopicId) -> bool:
        return topic_id.type == self._topic_type   # 按 type 匹配

    def map_to_agent(self, topic_id: TopicId) -> AgentId:
        return AgentId(type=self._agent_type, key=topic_id.source)

8. 功能开发场景动态过程

功能开发可建模为 planner、developer、tester、reviewer 的 team。每个 agent 有不同 system message、工具和记忆；manager 负责选择发言者并判断终止。

flowchart TD Start[用户提交功能开发任务] --> Init[构建 agents / tools / memory / team] Init --> Run[team.run_stream] Run --> Broadcast[Manager 广播任务] Broadcast --> Select[选择 speaker] Select --> AgentRun[Agent 处理消息] AgentRun --> Memory[注入短期上下文 + 长期记忆] Memory --> LLM[调用 LLM] LLM --> HasTool{是否请求工具?} HasTool -- 是 --> Tool[执行文件/代码/测试/MCP工具] Tool --> ToolResult[结果写回上下文] ToolResult --> More{继续工具轮?} More -- 是 --> LLM More -- 否 --> Finalize[反思或汇总] HasTool -- 否 --> Finalize Finalize --> Publish[发布 agent response] Publish --> Term{满足终止条件?} Term -- 否 --> Select Term -- 是 --> Result[输出 TaskResult] Result --> Persist[可选保存长期记忆]

推荐用明确终止条件防止循环，例如 reviewer 输出后结束、文本提及 TERMINATE、最大消息数或最大轮数。
