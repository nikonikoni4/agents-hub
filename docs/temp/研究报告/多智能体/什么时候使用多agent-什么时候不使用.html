何时使用多 Agent 系统

单 Agent 处理大多数工作流已足够，但多 Agent 架构能在特定场景释放额外价值。关键在于识别这些场景。

为什么从单 Agent 开始

每个额外的 Agent 都是一个潜在故障点、一组需要维护的 Prompt、一个意外行为的来源。

单 Agent

  • 一个上下文，无交接损耗
  • Prompt 维护成本低
  • Token 消耗可控
  • 调试简单，因果链清晰

多 Agent

  • 每次交接都丢失上下文
  • 多套 Prompt 需同步维护
  • 协调消息消耗额外 Token
  • 调试困难，因果链分散

多 Agent 擅长的三种场景

多 Agent 架构应保留给那些能提供明确收益、足以证明额外成本合理的场景。

①

上下文保护

Context Protection

当子任务产生大量上下文但大部分与主任务无关时，子 Agent 提供隔离，每个 Agent 在自己干净的上下文中工作。

适用条件
  • 子任务产生 >1000 Token 上下文
  • 大部分信息与主任务无关
  • 查找/检索操作需要过滤
②

并行化

Parallelization

多个 Agent 并行运行，探索比单 Agent 更大的搜索空间。核心收益是覆盖广度，而非速度。

适用条件
  • 任务可分解为独立研究路径
  • 需要搜索大量信息空间
  • 子任务之间无需共享上下文
③

专业化

Specialization

不同任务受益于不同的工具集、系统提示或领域专业知识。聚焦的 Agent 比全能 Generalist 更可靠。

三种专业化
  • 工具集：20+ 工具时拆分
  • 系统提示：行为模式冲突时拆分
  • 领域知识：深度上下文需隔离

以上下文为中心的分解

最关键的架构决策：如何划分 Agent 之间的工作。按上下文边界分解，而非按工作类型。

有效的分解边界

上下文可以真正隔离
  • 独立研究路径（亚洲市场 vs 欧洲市场）
  • 有清晰接口的独立组件（前后端）
  • 黑盒验证（只需运行测试并报告）

有问题的分解边界

上下文大量重叠
  • 同一工作的顺序阶段（规划→实现→测试）
  • 紧密耦合的组件（需频繁来回沟通）
  • 需要共享状态的工作（需频繁同步理解）

两种分解方式对比

按工作类型分解（常适得其反）
Agent A 写功能 → 交接 → Agent B 写测试 → 交接 → Agent C 审查代码
每次交接丢失上下文，协调 Token > 执行 Token
按上下文边界分解（通常有效）
Agent A：功能 + 它的测试（已拥有上下文）
Agent B：独立模块 + 它的测试
仅在上下文可真正隔离时才拆分
05

验证子 Agent 模式

跨领域始终表现良好的一种模式。验证天然需要最少的上下文传递——验证者是黑盒测试。

1
主 Agent 完成工作

实现功能、生成内容、执行任务

→
2
派发验证子 Agent

传递产出物 + 成功标准 + 验证工具

→
3
黑盒验证

不需要理解构建过程，只判断是否达标

→
4
通过 / 重试

通过则继续；失败则将问题注入下次尝试

「早胜问题」与缓解策略

验证者跑了一两个测试就宣布成功——最常见的失败模式。

具体标准
「运行完整测试套件并报告所有失败」而非「确保它能用」
全面检查
要求验证者测试多个场景和边界情况
负面测试
指示验证者尝试应该失败的输入并确认它们确实失败
显式指令
「你必须运行完整测试套件才能标记为通过」——不可或缺
06

决策清单

在引入多 Agent 复杂性之前，确认以下三点。

□
存在真正的约束 — 多 Agent 能解决的：上下文限制、并行化机会、或专业化需求
□
按上下文分解，非按工作类型 — 按所需上下文分组工作，而非按工作性质
□
存在清晰的验证点 — 子 Agent 可以在不需要完整上下文的情况下验证工作
「从最简单的可行方案开始，只在有证据支持时才增加复杂性。」
