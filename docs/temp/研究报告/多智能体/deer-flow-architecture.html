项目概览

DeerFlow 2.0 是一次彻底重写。它把 sub-agents、memory 和 sandbox 组织在一起，再配合可扩展的 skills，让 agent 可以完成几乎任何事情。

技术栈

  • 后端: Python 3.12+, FastAPI, LangGraph, LangChain
  • 前端: Next.js, TypeScript
  • 数据库: SQLite / PostgreSQL
  • 部署: Docker, Nginx

核心特性

  • Skills 与 Tools 系统
  • Sub-Agents 并行执行
  • Sandbox 与文件系统隔离
  • Context Engineering
  • 长期记忆管理

支持的IM渠道

  • 飞书 (Feishu) - 支持流式
  • 企业微信 (WeCom) - 支持流式
  • Slack, Telegram, Discord
  • 钉钉 (DingTalk), 微信公众号

Nginx 与 Gateway

基础概念

Nginx — 反向代理 / 流量调度

  • 是什么：Web 服务器 + 反向代理
  • 做什么：统一接收所有请求，根据 URL 路径转发给不同后端服务
  • 类比：大楼前台 — 所有访客先到前台，前台引导到对应办公室
  • 核心价值：客户端只需知道一个入口地址，无需关心后端有几个服务、各自在哪个端口

Gateway — API 网关

  • 是什么：后端 API 的统一入口（通常是一个 Web 应用）
  • 做什么：处理认证、鉴权、CSRF 防护、路由分发等公共逻辑
  • 类比：公司门卫 — 验证身份、检查权限，再放行到具体部门
  • 核心价值：公共逻辑只写一次，所有路由自动受益

两者的分工

Nginx 管基础设施（端口转发、负载均衡、SSL）， Gateway 管业务逻辑（认证、鉴权、路由到具体处理函数）。 两者配合，形成完整的请求处理链。

DeerFlow 中的 Nginx 配置

DeerFlow 的 Nginx 配置在 docker/nginx/nginx.local.conf，核心逻辑非常简单：以 /api/ 开头的请求转给 Gateway，其他请求转给前端。

# 定义后端服务地址
upstream gateway { server 127.0.0.1:8001; }
upstream frontend { server 127.0.0.1:3000; }

server {
  listen 2026;  // 统一入口端口

  # /api/langgraph/* → 去掉前缀后转给 Gateway
  location /api/langgraph/ {
    rewrite ^/api/langgraph/(.*) /api/$1 break;
    proxy_pass http://gateway;
  }

  # /api/models → 转给 Gateway
  location /api/models { proxy_pass http://gateway; }

  # /api/memory → 转给 Gateway
  location /api/memory { proxy_pass http://gateway; }

  # /api/skills → 转给 Gateway
  location /api/skills { proxy_pass http://gateway; }

  # /api/threads/*/uploads → 转给 Gateway（允许 100M 上传）
  location ~ ^/api/threads/[^/]+/uploads {
    proxy_pass http://gateway;
    client_max_body_size 100M;
  }

  # 兜底：其他 /api/* 全转给 Gateway
  location /api/ { proxy_pass http://gateway; }

  # 其他所有请求 → 转给前端
  location / { proxy_pass http://frontend; }
}

DeerFlow 中的 Gateway 实现

Gateway 是一个 FastAPI 应用（backend/app/gateway/app.py），核心职责：注册中间件 + 注册路由。

def create_app() -> FastAPI:
  app = FastAPI(title="DeerFlow API Gateway")

  # 注册中间件（请求进来时按顺序执行）
  app.add_middleware(AuthMiddleware)  # 1. 验证身份
  app.add_middleware(CSRFMiddleware)  # 2. CSRF 防护
  app.add_middleware(CORSMiddleware) # 3. 跨域处理

  # 注册路由（URL → 处理函数）
  app.include_router(auth.router)       # /api/v1/auth/*
  app.include_router(thread_runs.router) # /api/threads/*/runs/*
  app.include_router(models.router)     # /api/models/*
  app.include_router(mcp.router)        # /api/mcp/*
  app.include_router(memory.router)     # /api/memory/*
  app.include_router(skills.router)     # /api/skills/*
  # ... 还有 agents, uploads, artifacts 等路由
  return app

认证中间件示例

Gateway 的 AuthMiddleware（auth_middleware.py）是"门卫"的核心实现：每个请求进来先过这道关。

class AuthMiddleware:
  async def dispatch(self, request, call_next):
    # 1. 公开路径（如 /health、/login）直接放行
    if _is_public(request.url.path):
      return await call_next(request)

    # 2. 没有 access_token cookie → 返回 401
    if not request.cookies.get("access_token"):
      return JSONResponse(status_code=401)

    # 3. 验证 JWT，解析出用户
    user = await get_current_user_from_request(request)

    # 4. 把用户信息挂到 request 上，下游都能用
    request.state.user = user
    return await call_next(request)

Middleware 机制详解：DeerFlow 的"中间链条"

什么是 Middleware？

Middleware 是包裹在 Agent 循环外部的拦截层。DeerFlow 的 Agent 本质是一个单节点 LangGraph 图，运行着 model call → tool execution → model call → ... 的循环。Middleware 不参与业务决策，而是在这个循环的每个阶段注入横切关注点（cross-cutting concerns）——错误处理、沙箱管理、记忆注入、安全审计等。

可以把 Agent 循环想象成一条流水线，Middleware 就是流水线两侧的质检工位：产品（消息）在流水线上流动，每个工位检查、修改或记录产品状态，但工位本身不决定产品是什么。

Middleware 挂钩点

每个 Middleware 可以实现以下钩子，在 Agent 循环的不同阶段插入逻辑：

记忆提取什么？

记忆不是完整聊天记录，而是从可复用上下文中提取出的结构化资料。输入会先过滤掉 tool call 中间消息、纯上传文件消息和临时上传路径，只保留用户输入与最终 assistant 回复。

摘要型记忆

  • workContext — 用户工作背景、项目、技术栈
  • personalContext — 语言偏好、沟通风格、个人偏好
  • topOfMind — 近期关注的多个主题和活跃任务
  • recentMonths — 近几个月活动和技术探索
  • earlierContext — 更早但仍有价值的历史模式
  • longTermBackground — 长期稳定背景和核心能力

事实型记忆

  • preference — 偏好的工具、风格、处理方式
  • knowledge — 掌握的技术和领域知识
  • context — 项目、团队、角色、环境背景
  • behavior — 工作习惯、沟通习惯、决策模式
  • goal — 目标、学习方向、项目意图
  • correction — 用户纠正过的误解和以后应避免的错误

记忆为什么放在 HumanMessage，而不是 system prompt？

当前实现把记忆放进隐藏的 HumanMessage，不是为了把它当作用户原文，而是为了保持静态 system prompt 不变，提升 provider prefix cache 的命中率。如果把用户记忆直接拼进 system prompt，每个用户、每个 Agent、每次记忆变化都会改变 prompt 前缀，缓存复用会明显变差。

位置 用途 取舍
静态 system prompt 长期规则、角色、工具使用规范 保持稳定，利于 prefix cache；适合放强约束
隐藏 system-reminder HumanMessage 用户记忆、当前日期等运行时上下文 每个会话可变；语义上提醒模型，但角色层级仍是 user-role 内容
// DynamicContextMiddleware 注入的隐藏消息形态
<system-reminder>
  <memory>
    User Context:
    - Work: ...
    - Current Focus: ...

    History:
    - Recent: ...

    Facts:
    - [preference | 0.95] User prefers concise technical explanations
  </memory>

  <current_date>2026-05-25, Monday</current_date>
</system-reminder>

<system-reminder> 不是模型 API 的正式角色，也不是标准 XML 协议。它是一种提示标签约定，用来告诉模型“这段是运行时提醒，不是用户请求本身”。Claude 系工具中经常用类似标签注入当前日期、文件状态、工具规则或上下文提醒。真正的权限层级仍由消息 role 决定：API 层面的 system 高于 user，因此这类标签适合放记忆和环境提示，不适合放不可违反的安全策略。

Middleware 与 MetaGPT Action SOP 的对比

两者都是预定义、可重复执行的线性管道，但解决的问题层级不同。
维度 DeerFlow Middleware 链 MetaGPT Action SOP
组成单元 Middleware（基础设施关注点） Action（业务领域步骤）
典型节点 ErrorHandling, Memory, Guardrail, Sandbox WritePRD, Design, WriteCode, SummarizeCode
编排方式 线性链，append 顺序执行 线性链（BY_ORDER），顺序执行
关注层级 横切关注点 基础设施：怎么跑 纵向业务流 领域流程：做什么
产出物 无独立产出，修饰 Agent 行为 每步产出 Message / Artifact
与 LLM 的关系 Middleware 本身不调用 LLM，只拦截 LLM 的输入/输出 Action 内部调用 LLM 完成具体业务
可复用性 跨场景通用，可插拔组合 按角色预定义，绑定特定 SOP
类比 流水线两侧的质检工位 流水线上的加工工位

MetaGPT 的 Action SOP 回答的是 "先做什么、再做什么"——写 PRD、做设计、写代码，每一步都产出业务制品。DeerFlow 的 Middleware 链回答的是 "在做任何事之前/之后，环境需要准备什么、检查什么、记录什么"——沙箱就绪了吗？记忆注入了吗？异常兜住了吗？

两者是同一棵编排树上的两个不同分支，共享"预定义线性管道"这个骨架。它们可以叠加：你可以在 MetaGPT 的某个 Action 内部跑一个 DeerFlow 风格的 Middleware 链，也可以在 DeerFlow 的 Agent 循环里调用一个 MetaGPT 风格的 Action 序列。

Step 1-2：用户请求 → 前端 → Nginx

用户在 Next.js 界面输入任务。前端将消息序列化为 RunCreateRequest，发送 POST /api/threads/{thread_id}/runs/stream。Nginx 匹配 location /api/，将请求转发到 Gateway :8001。

Step 3：Gateway 认证层

AuthMiddleware 检查 access_token cookie → 验证 JWT → 解析出 user_id，挂到 request.state.user。CSRFMiddleware 验证 Double Submit Cookie。CORSMiddleware 处理跨域。

Step 4：Router → start_run()

thread_runs.py::stream_run() 接收请求，调用 start_run()：
① run_mgr.create_or_reject() → 创建 RunRecord（run_id, thread_id, status=pending）
② normalize_input() → 将 dict 消息转为 LangChain Message 对象
③ build_run_config() → 注入 thread_id、recursion_limit=100
④ merge_run_context_overrides() → 注入 model_name、subagent_enabled=true 等
⑤ asyncio.create_task(run_agent(...)) → 后台启动 Agent
⑥ 返回 StreamingResponse，绑定 SSE consumer

Step 5-6：run_agent() → make_lead_agent()

run_agent() 初始化 RunJournal、标记 status=running、快照 checkpoint（用于回滚）、构建 Runtime context、注入 LangChain 回调。
然后调用 make_lead_agent(config)：解析模型名 → 组装 Middleware 链 → 获取 Tools → 调用 create_agent() 编译为 LangGraph StateGraph。

Step 7：Middleware 链执行（before_agent 阶段）

Agent 循环启动前，before_agent 挂钩按顺序执行：
① ThreadDataMiddleware → 创建 /data/users/u123/threads/abc/ 工作目录
② UploadsMiddleware → 扫描上传文件列表注入 state
③ SandboxMiddleware → 获取沙箱实例，写入 sandbox_id
...
⑧ DynamicContextMiddleware → 从记忆系统读取用户偏好，注入 <memory> 标签 + 当前日期到首条 HumanMessage 前

Step 8：Lead Agent LLM 决策

LLM 收到系统提示 + 注入的记忆上下文 + 用户消息，分析后决定：
→ 调用 task("实现后端登录API，包括JWT认证、用户表、登录注册接口", subagent_type="general-purpose")
→ 调用 task("实现前端登录页面，包括表单、验证、API对接", subagent_type="general-purpose")
两个 tool_call 在同一轮 AIMessage 中发出。

Step 9：Middleware 链执行（wrap_tool_call 阶段）

每个 tool_call 经过 wrap_tool_call 挂钩：
① SubagentLimitMiddleware → 检查并行数（默认上限 3），截断超额调用
② ToolErrorHandlingMiddleware → try/except 包裹工具执行，异常转为错误 ToolMessage
③ SandboxAuditMiddleware → 记录沙箱操作审计日志
④ GuardrailMiddleware → 可选的工具调用授权检查

Step 10：task_tool() → SubAgent 执行

task_tool() 为每个子任务：
① 验证 subagent_type，提取父级上下文（sandbox_state, thread_data, trace_id）
② 创建 SubagentExecutor，提交到线程池（3 workers）
③ execute_async() → 在独立 asyncio 事件循环中运行 _aexecute()
④ SubAgent 内部也有自己的 Middleware 链（ThreadData + Sandbox + ErrorHandling 等基础设施层）
⑤ Lead Agent 的 task_tool 轮询每 5 秒检查一次，通过 get_stream_writer() 将 SubAgent 进度推送给前端 SSE
⑥ 两个 SubAgent 并行执行，各自独立的沙箱和工作目录

Step 11：结果汇总 → 最终响应

两个 SubAgent 完成 → task_tool 返回 "Task Succeeded. Result: ..." 作为 ToolMessage。
Lead Agent LLM 收到两个 ToolMessage，生成汇总响应："登录功能开发完成，后端 API 包含...前端页面包含..."

Step 12：收尾 → SSE 关闭

after_agent 挂钩执行：
① MemoryMiddleware → 将本次对话过滤后入队记忆更新（30s 去抖后异步写入）
② TitleMiddleware → 自动生成会话标题"用户登录功能开发"
run_agent() → flush journal、持久化 token 用量、同步标题到 thread_meta、发布 end 事件
sse_consumer() → 流结束，HTTP 连接关闭

2.6 Checkpoint — 断点续传的保障

LangGraph 的 Checkpointer 在每一步（每次 LLM 调用、每次工具执行）后自动保存 ThreadState 快照到数据库。这意味着：
  • 如果 run_agent 崩溃或超时，可以从最近的 checkpoint 恢复
  • 用户可以在任意步骤中断后继续（Human-in-the-loop）
  • run_agent 启动时会先快照 pre-run checkpoint，用于 rollback

LangGraph 在 DeerFlow 中的真正角色

核心结论：DeerFlow 用的是 LangGraph 的运行时基础设施，不是它的图编排能力。Lead Agent 在 LangGraph 层面只有一个节点，SubAgent 不是图节点，是工具函数内部启动的独立线程。

为什么单节点还要用 LangGraph？

如果不用 LangGraph，以下能力全部要自己实现：

LangGraph 提供的能力 不用 LangGraph 要自己写
Checkpoint 自动持久化 状态快照 + 存储 + 恢复逻辑
Interrupt / Resume（Human-in-the-loop） 中断点管理 + 恢复上下文重建
add_messages 状态合并器 消息去重、追加、替换的手动处理
SSE 流式输出（values / updates / messages 三种模式） 生产者-消费者 + 序列化 + 断线重连
create_agent() 标准 ReAct 循环 while 循环 + tool_call 解析 + 错误重试
Middleware 钩子系统 before / after 拦截器链
Checkpointer 插件化（SQLite / Postgres） 适配不同存储后端

LangGraph 在这里不是"图编排引擎"，而是"带状态持久化的 Agent 运行时"。类比：你买车用来代步，但车里有空调、音响、倒车雷达——你用的是这些功能，不是因为"需要一辆有四个轮子的东西"。

Checkpoint 什么时候保存？

每个节点每次执行完毕后保存一次。对于单节点 ReAct 循环，就是每次 LLM 返回后保存：

// 一次包含 2 次 LLM 调用的 run，产生 3 个 checkpoint

第 1 轮: LLM → tool_calls = [task("后端"), task("前端")]
  ── checkpoint ① ──  // 快照: messages + sandbox + artifacts

第 2 轮: 工具执行 → 两个 ToolMessage 加入 state
  ── checkpoint ② ──  // 快照: messages 多了 2 条 ToolMessage

第 3 轮: LLM → 最终 AIMessage（无 tool_call）→ 结束
  ── checkpoint ③ ──  // 快照: 终态

Checkpoint 存了什么？

  • ThreadState 全量快照：messages、sandbox、artifacts、todos、title
  • 元数据：thread_id、step 序号、时间戳
  • 写入位置：SQLite / PostgreSQL（通过 Checkpointer 插件）

Checkpoint 的用途

  • 崩溃恢复：run_agent 崩溃后从最近 checkpoint 恢复
  • Human-in-the-loop：用户暂停后可以从断点继续
  • 回滚：run_agent 启动时快照 pre-run checkpoint，失败可回滚
  • 会话历史：thread 恢复时加载最近 checkpoint 的 state

关键设计模式

1. Harness/App 分层

  • persistence 属于 harness 层（可发布包）
  • app/gateway 属于应用层
  • 依赖单向：app 导入 deerflow，反之禁止
  • CI 测试强制执行边界

2. 双轨持久化

  • LangGraph Checkpointer: 管理图执行状态
  • 应用 ORM: 管理业务元数据
  • 共享物理数据库但代码完全独立

3. Repository 模式 + 用户隔离

  • 所有数据访问通过 Repository 抽象
  • user_id 通过 ContextVar 自动传播
  • 实现行级数据隔离

4. 内存优先 + Store 兜底

  • RunManager 在内存维护活跃运行状态
  • 同步持久化到 Store
  • 支持进程重启恢复

5. 事件溯源

  • RunEventStore 以 append-only 方式记录
  • 按 seq 全局递增
  • 支持分页查询和审计

6. 消息总线解耦

  • MessageBus 完全解耦 Channel 和 Agent
  • Channel 不知道哪个 Agent 处理消息
  • Agent 不知道消息来自哪个平台
