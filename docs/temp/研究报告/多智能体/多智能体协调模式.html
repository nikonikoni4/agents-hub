多智能体协调模式
五种方法及其适用场景

五种多智能体协作模式的运作机制、优缺点与适用场景——以及何时从一种模式演进到另一种。建议从最简单的模式入手，观察不足后再演进。

1

生成器-验证器

Generator-Verifier

最简单的多智能体模式，也是应用最广泛的。生成器接收任务产生输出，提交给验证器评估。验证器检查是否符合标准：通过则接受，不通过则返回反馈让生成器重新尝试。循环直到通过或达到最大迭代次数。

优势

  • 实现简单，调试方便
  • 输出质量有明确保障
  • 反馈循环可逐步改进
  • 评估标准可量化

局限

  • 验证器质量取决于标准定义
  • 生成与验证技能可能难以分离
  • 迭代可能停滞不收敛
  • 需要最大迭代限制兜底

适用场景

  • 代码生成（一个写代码，一个写测试并运行）
  • 事实核查与合规性验证
  • 基于评分标准的质量评估
  • 客服邮件回复的准确性审核
2

协调器-子代理

Orchestrator-Subagent

层级结构定义此模式。一个代理充当团队领导，规划工作、分配任务并汇总成果。子代理处理具体职责后汇报结果。Claude Code 即采用此模式——主代理编写代码、编辑文件，需要时派发子代理在后台搜索代码库或调查独立问题。

优势

  • 任务分解清晰，职责明确
  • 协调器保持全局视野
  • 子代理可并行执行
  • 实现复杂度适中

局限

  • 协调器成为信息瓶颈
  • 子代理间信息传递需经过协调器
  • 顺序执行时无速度优势
  • 关键细节在多次传递中易丢失

适用场景

  • 自动代码审查（安全、测试覆盖、风格、架构）
  • 任务分解明确、子任务相互依赖少的场景
  • 需要保持全局一致性的多步骤工作流
3

代理团队

Agent Teams

当工作分解为可长时间独立进行的并行子任务时，协调器-子代理模式会显得过于受限。代理团队中，协调器生成多个持久化的工作代理，工人从共享队列中认领任务、自主完成多步骤工作后标记完成。工人不会在任务间被重置，持续积累领域知识。

优势

  • 工人持续运行，积累领域上下文
  • 真正并行，适合大规模任务
  • 自主完成多步骤，无需逐步指挥
  • 随时间提升性能

局限

  • 独立性要求高，难以共享中间发现
  • 完成检测更困难（耗时不同）
  • 共享资源可能冲突（编辑同一文件）
  • 需要任务分区和冲突解决机制

适用场景

  • 大型代码库迁移（每个工人负责一个服务）
  • 独立分区的数据处理任务
  • 需要持续多步骤、无需频繁协调的并行工作
4

消息总线

Message Bus

当代理数量增多、交互模式复杂化时，直接协调变得难以管理。消息总线引入共享通信层：代理通过「发布」和「订阅」两种原语交互。代理订阅感兴趣的主题，路由器将匹配消息投递。新代理可随时接入，无需重新连接现有结构。

优势

  • 松耦合，新代理可随时接入
  • 事件驱动，工作流灵活涌现
  • 各团队可独立开发部署代理
  • 路由可扩展

局限

  • 事件级联后追踪困难
  • 调试比顺序执行更复杂
  • 路由准确性至关重要（误分类则静默失败）
  • LLM 路由器引入新故障模式

适用场景

  • 安全运营自动化（告警分类 → 调查 → 响应）
  • 事件驱动流水线，工作流从事件中涌现
  • 代理生态系统持续增长的场景
5

共享状态

Shared State

前述模式均由中心节点管理信息流。共享状态消除中间人：代理通过持久化存储直接读写协作，无中央协调器。工作从初始化种子数据开始，满足终止条件（时间限制、收敛阈值、充分性判断）时结束。存储逐渐演变为知识库。

优势

  • 消除单点故障
  • 代理可实时看到彼此发现
  • 知识库持续演进
  • 一个代理停止不影响其他

局限

  • 无显式协调，可能重复工作
  • 反应式循环消耗资源不收敛
  • 结果不可预测
  • 需要明确的终止条件（时间/收敛/充分性）

适用场景

  • 多代理研究综合系统（学术文献 + 行业报告 + 专利 + 新闻）
  • 需要代理实时共享发现的协作场景
  • 消除单点故障为优先目标的系统

模式演进指南

对比维度 选择 A 选择 B 关键判断
协调器-子代理 vs 代理团队 子任务短、聚焦、输出明确 子任务需持续多步骤工作 工人是否需要跨调用保持状态？
协调器-子代理 vs 消息总线 步骤序列事先已知 工作流从事件中涌现、不可预测 条件逻辑是否在不断膨胀？
代理团队 vs 共享状态 代理在独立分区工作，不交互 代理需要实时看到彼此发现 代理是否需要彼此的中间成果？
消息总线 vs 共享状态 代理响应事件触发动作 代理基于累积知识持续工作 是离散事件流还是知识积累？
