# Agent Bridge 框架调研结果

## 1. 框架结构概述

Agent Bridge 是 agents-hub 系统的**纯执行层模块**，负责调用不同 AI 平台的 CLI 工具，并将各平台的原始输出解析为统一格式。

### 目录结构
```
agents_hub/agent_bridge/
├── __init__.py          # 模块初始化，导出核心类
├── bridge.py            # 核心桥接器，统一调用接口
├── models.py            # 数据模型定义（StreamEvent, AgentResult）
├── protocols.py         # 协议定义（Executor, Parser）
├── exceptions.py        # 异常类型定义
├── executors/           # 执行器实现
│   ├── claude.py        # Claude CLI 执行器
│   ├── codex.py         # Codex CLI 执行器
│   ├── docker_claude.py # Docker 沙箱下的 Claude 执行器
│   ├── docker_codex.py  # Docker 沙箱下的 Codex 执行器
│   └── docker_base.py   # Docker 执行器基类
├── parsers/             # 解析器实现
│   ├── claude.py        # Claude 输出解析器
│   └── codex.py         # Codex 输出解析器
└── docker/              # Docker 沙箱管理
    ├── manager.py       # Docker 容器管理器
    ├── container.py     # 容器操作封装
    └── models.py        # Docker 相关数据模型
```

## 2. 核心组件分析

### 2.1 协议定义（protocols.py）

定义了两个核心协议接口：

1. **Executor 协议**：负责启动 CLI 并返回原始输出流
   - 方法：`execute(prompt, config, session_id, cwd, fork_from) -> AsyncIterator[str]`
   - 职责：构建 CLI 命令、启动子进程、返回原始 JSON 字符串流

2. **Parser 协议**：负责解析原始输出为统一格式
   - 方法：`parse_event(raw_line) -> StreamEvent | None`
   - 职责：解析单行 JSON 事件，转换为统一的 StreamEvent 格式

### 2.2 数据模型（models.py）

#### StreamEvent（流式事件）
```python
@dataclass
class StreamEvent:
    type: AgentEventType  # 事件类型枚举
    content: dict         # 具体数据
    session_id: str       # 会话 ID
    timestamp: str        # 时间戳
    agent_name: str       # agent 名称（由 Bridge 填充）
    platform: AgentPlatform  # 平台类型（由 Bridge 填充）
    role_type: RoleType   # 角色类型（由 Bridge 填充）
```

#### AgentResult（完整结果）
```python
@dataclass
class AgentResult:
    text: str             # 完整文本
    session_id: str       # 会话 ID
    timestamp: str        # 时间戳
    agent_name: str       # agent 名称
    platform: AgentPlatform  # 平台类型
    role_type: RoleType   # 角色类型
    usage: dict | None    # token 使用统计
    cwd: str | None       # 工作目录
    modified_files: list[FileMetadata] | None  # 修改的文件列表
    git_diff_range: str | None  # Git diff 范围
```

#### AgentEventType（事件类型枚举）
- `INIT`：会话开始元数据
- `TEXT_DELTA`：文本增量（流式输出的主要内容）
- `TOOL_USE`：工具调用
- `TURN_COMPLETE`：回合完成（包含 token 使用统计）
- `RESULT`：完整结果（非流式输出）

### 2.3 桥接器（bridge.py）

`AgentBridge` 类是核心，提供三种调用接口：

1. **`execute_stream()`**：流式执行，逐事件 yield StreamEvent
2. **`execute()`**：非流式执行，返回完整 AgentResult
3. **`bare_claude_call()`**：快速 LLM 调用，使用 bare 角色配置

#### 初始化逻辑
- 创建 Claude/Codex 的 Executor 和 Parser 实例
- 创建 RoleManager 实例
- 初始化 bare 角色配置（`_init_bare_config()`）

#### 数据流
```
用户调用 → Bridge.execute_stream()
  → 根据 platform 选择 Executor + Parser
  → Executor 启动 CLI 子进程，返回原始 JSON 流
  → Parser 逐行解析为统一 StreamEvent
  → yield 给调用方
```

### 2.4 执行器实现

#### ClaudeExecutor（claude.py）
- 命令：`claude --print --verbose --output-format stream-json --include-partial-messages`
- 支持参数：`--resume`（恢复会话）、`--fork-session`（fork 会话）、`--append-system-prompt`（系统提示词）
- 环境变量：`CLAUDE_CONFIG_DIR`（取自 `RoleConfig.work_root`）

#### CodexExecutor（codex.py）
- 命令：`codex exec --json`（新会话）或 `codex exec resume --json <session_id>`（恢复会话）
- 支持参数：`-C`（工作目录）、`-c instructions=<system_prompt>`（系统提示词）
- 环境变量：`CODEX_HOME`（取自 `RoleConfig.work_root`）

### 2.5 解析器实现

#### ClaudeParser（parsers/claude.py）
- 解析 Claude CLI 的流式 JSON 输出
- 支持事件类型：`stream_event`（文本增量）、`system`（系统事件）
- 工具调用解析：默认关闭（`ENABLE_TOOL_USE_PARSING = False`）

#### CodexParser（parsers/codex.py）
- 解析 Codex CLI 的流式 JSON 输出
- 支持事件类型：`thread.started`、`item.completed`、`turn.completed`
- 会话标识：从 `thread_id` 获取

## 3. 设计模式总结

### 3.1 执行器-解析器分离模式
- **Executor**：负责 CLI 命令构建和进程管理
- **Parser**：负责输出格式解析和事件转换
- **Bridge**：负责组件组装和流程协调

### 3.2 协议驱动设计
- 使用 Python Protocol 定义接口契约
- 新平台只需实现 Executor 和 Parser 协议
- 通过 Bridge 注册即可使用

### 3.3 统一事件模型
- 所有平台输出转换为统一的 StreamEvent 格式
- 事件类型枚举确保类型安全
- 由 Bridge 统一填充 agent_name、platform、role_type

### 3.4 三种调用接口
1. **流式接口**：适用于人机交互场景（实时显示）
2. **非流式接口**：适用于 A2A 调用场景（主 Agent 调用子 Agent）
3. **快速调用接口**：适用于内部快速 LLM 调用（不涉及角色业务）

## 4. 重点问题分析

### 4.1 流式输出处理
- **当前实现**：所有平台输出都转换为流式事件，通过 `execute_stream()` 逐事件 yield
- **非流式转换**：`execute()` 内部拼接所有 `text_delta` 事件文本，收集 `usage` 统计
- **设计特点**：流式是基础，非流式是薄包装

### 4.2 系统提示词添加
- **Claude**：通过 `--append-system-prompt` 参数注入
- **Codex**：通过 `-c instructions=<system_prompt>` 参数注入
- **设计特点**：系统提示词作为可选参数传递，不存储在 RoleConfig 中

### 4.3 MCP 集成
- **当前状态**：未在代码中发现 MCP 集成逻辑
- **设计思路**：MCP 可能作为工具（tool）集成，通过 CLI 的工具调用机制实现
- **潜在方案**：在 Executor 中配置 MCP 工具，通过 CLI 参数传递

## 5. OpenCode 集成建议

### 5.1 集成步骤
1. **添加平台枚举**：在 `AgentPlatform` 中添加 `OPENCODE = "opencode"`
2. **添加命令常量**：在 `types.py` 中添加 `OPENCODE_COMMAND`
3. **实现执行器**：创建 `OpenCodeExecutor` 类，实现 Executor 协议
4. **实现解析器**：创建 `OpenCodeParser` 类，实现 Parser 协议
5. **注册到 Bridge**：在 `AgentBridge.__init__()` 中注册新的执行器和解析器

### 5.2 执行器实现要点
- 研究 OpenCode CLI 的命令行参数和输出格式
- 实现 `execute()` 方法，返回原始 JSON 字符串流
- 支持会话恢复（`session_id`）和系统提示词注入

### 5.3 解析器实现要点
- 分析 OpenCode CLI 的输出 JSON 结构
- 实现 `parse_event()` 方法，转换为统一的 StreamEvent
- 处理文本增量、工具调用、会话完成等事件类型

### 5.4 流式输出处理建议
- 如果 OpenCode 支持流式 JSON 输出，直接解析
- 如果只支持非流式输出，需要在 Executor 中实现缓冲和模拟流式
- 建议优先支持流式输出，保持与现有框架的一致性

### 5.5 系统提示词集成建议
- 研究 OpenCode CLI 的系统提示词注入方式
- 在 `_build_command()` 方法中添加相应参数
- 确保与现有 Claude/Codex 的集成方式一致

### 5.6 MCP 集成建议
- 如果 OpenCode 支持 MCP 工具，可在 Executor 中配置
- 通过 CLI 参数传递 MCP 工具配置
- 考虑与现有工具调用机制的兼容性

## 6. 总结

Agent Bridge 框架采用清晰的执行器-解析器分离架构，通过协议驱动设计实现了良好的扩展性。添加新的 agent 类型（如 OpenCode）只需实现对应的 Executor 和 Parser，并注册到 Bridge 即可。

框架的核心优势：
1. **统一接口**：所有平台通过相同的接口调用
2. **流式优先**：流式输出是基础，非流式是薄包装
3. **易于扩展**：新平台只需实现两个协议类
4. **配置灵活**：通过 RoleConfig 管理平台特定配置

对于 OpenCode 集成，建议遵循现有模式，重点研究其 CLI 输出格式和系统提示词注入方式，确保与现有框架的兼容性。
