设计原则

Three Core Principles

本架构围绕三个核心原则设计，确保多 Agent 系统的消息传递既安全又高效。

🛡

避免越权访问

Agent 不能直接访问其他 Agent 的方法和状态。每个 Agent 只持有 MessageRouter 引用，无法窥探其他 Agent 的内部。

📋

按需提供上下文

每个 Agent 只获取与其职责相关的上下文信息，通过 GroupChatContext 过滤无关消息，减少 token 消耗。

🎯

点对点优于广播

消息精确路由到目标 Agent 的私有队列，而非广播后过滤。真正的点对点通信，无信息泄露。

核心组件详解

Core Components

3.1 Message（消息）

消息是架构的基本通信单元，承载内容和路由信息。每个消息都有明确的发送者、接收者和可选的调用 ID。

Message Class
classDiagram class Message { +str content +str send_to +str sent_from +str call_id +datetime timestamp }

@ 符号的双重语义

User 的 @：有路由作用。系统解析 @Agent A 后直接路由到 Agent A，绕过 Manager。

Agent 的 @：无路由作用，仅用于上下文澄清。让其他 Agent 看到消息时知道任务对象是谁，避免误解。

3.2 AgentCallManager（调用管理器）

统一管理所有跨 Agent 的异步调用，提供状态追踪、生命周期管理和自动清理能力。

AgentCallManager Class
classDiagram class AgentCallManager { -dict _calls -Lock _lock -Logger _logger +create_call(send_from, send_to, content) str +get_call(call_id) AgentCall +update_status(call_id, status) +set_result(call_id, result) +set_error(call_id, error) +get_stats() dict -_cleanup_loop() -_cleanup_expired_calls() -_check_timeouts() -_log_call(call) -_search_call_in_log(call_id) } class AgentCall { +str call_id +str send_from +str send_to +str content +MessageType message_type +CallStatus status +datetime created_at +datetime started_at +datetime completed_at +AgentResult result +str error +str business_task_id +int timeout_seconds +is_timeout() bool +can_be_deleted() bool } class CallStatus { <> PENDING RUNNING COMPLETED FAILED TIMEOUT } class MessageType { <> TASK NOTIFICATION QUERY } AgentCallManager --> AgentCall AgentCall --> CallStatus AgentCall --> MessageType

call_id 生成策略

使用 UUID4 截断前 8 位（十六进制），在 100 个缓存下冲突概率约 0.00012%。使用 while 循环处理极小概率的连续冲突。短小精悍，易读易用。

3.3 MessageRouter（消息路由总线）

管理所有 Agent 的私有队列引用，负责点对点消息路由和历史记录。

MessageRouter Class
classDiagram class MessageRouter { -dict _agent_buffers -GroupChatContext _context +register(agent_name, buffer) +publish_message(msg) } class Agent { +str name +Queue message_buffer +GroupChatContext context -MessageRouter _bus +send_message(content, send_to) +run() -_process_message(msg) } Agent --> MessageRouter : 持有引用 MessageRouter --> Agent : 注册队列

MetaGPT 方式（双向引用）

Agent.rc.env.roles[name] 可以直接访问其他 Agent 的任何方法，存在越权风险。

本架构方式（单向引用）

Agent 只能调用 _bus.publish_message()，无法访问 _agent_buffers（私有变量），完全避免越权。

3.4 Agent（智能体）

每个 Agent 持有私有消息队列，从队列接收消息并处理，通过 MessageRouter 发送消息，通过 GroupChatContext 读取上下文。

Agent Message Processing

3.5 GroupChatContext（群聊上下文）

管理群聊历史、压缩历史和 Agent 调用状态，为每个 Agent 提供定制化上下文。

GroupChatContext Structure

上下文过滤规则：Agent 只看到与自己相关的消息，避免被无关信息干扰，减少 token 消耗。这不是"耦合"，而是"依赖注入"。

Call Lifecycle Management

状态流转图

每个 AgentCall 从创建到清理，经历明确的状态流转。超时由后台线程定期检查，自动标记为 TIMEOUT。

Call Status State Machine
stateDiagram-v2 [*] --> PENDING : create_call() PENDING --> RUNNING : Agent 开始处理 PENDING --> TIMEOUT : 超时检查 RUNNING --> COMPLETED : 执行成功 RUNNING --> FAILED : 执行失败 RUNNING --> TIMEOUT : 超时检查 COMPLETED --> [*] : 清理 FAILED --> [*] : 清理 TIMEOUT --> [*] : 清理

生命周期清理规则

基于状态和时间的智能清理策略，而非简单的 LRU 缓存。后台线程每 30 秒检查一次。

状态 MessageType 删除条件 保留时间 说明
PENDING 任意 不删除 - 等待执行，不能删除
RUNNING 任意 不删除 - 正在执行，不能删除
COMPLETED TASK 完成后 60 秒 60 秒 给发起者查询的时间
COMPLETED NOTIFICATION 立即删除 0 秒 不需要回复，立即清理
FAILED 任意 失败后 300 秒 5 分钟 便于调试
TIMEOUT 任意 超时后 300 秒 5 分钟 便于调试

关键设计要点

FIFO 保证：asyncio.Queue 保证消息按顺序处理。Manager 连续发送 3 个调用，Agent 按顺序执行。

异步解耦：发送者调用 call_agent() 立即返回 call_id，不等待执行完成。接收者从队列阻塞等待。发送者可以继续做其他事情。

消息记录：所有消息都记录到 GroupChatSession，包括 User 消息、Manager 调用、Agent 结果。写入时机：LLM 返回后、发送下一条消息前。

关键设计决策

Key Design Decisions

决策 1：为什么不用 MetaGPT 的双向引用？

MetaGPT 方式

Agent.rc.env → Environment
Environment.roles[name] → Agent
Agent 能通过 self.rc.env.roles[name] 访问其他 Agent 的任何方法，存在越权风险。

本架构方式

Agent._bus → MessageRouter
MessageRouter._agent_buffers[name] → Queue
Agent 只能调用 _bus.publish_message()，无法访问私有变量，完全避免越权。

决策 2：为什么不用 AutoGen 的公共 Buffer？

AutoGen 方式

GroupChat.message_buffer = []
所有 Agent 都能看到所有消息，依赖 Agent "自觉"不偷看，不符合"按需提供上下文"原则。

本架构方式

每个 Agent 有私有队列，MessageRouter 精确路由到目标队列。Agent 的队列里只有发给自己的消息，不需要过滤。

决策 3：为什么不用中心化暂存队列？

中心化队列方式

多了一层中转，增加延迟，需要后台任务维护，增加复杂度，无实际价值。

本架构方式

publish_message() 直接路由到私有队列，简单直接，无额外延迟，不需要后台任务。

决策 4：call_agent 立即返回 vs 等待结果

决策：采用异步设计，立即返回 call_id，不等待执行完成。

理由：真正的异步，调用者不阻塞，支持并发调用，符合现实场景。通过 query_call_status 工具查询结果。

决策 5：完成通知机制

决策：默认不自动通知，A 需要主动查询。可选支持自动通知。

理由：避免无限循环（A → B → A → B ...）。通过 MessageType 区分 TASK 和 NOTIFICATION，TASK 需要回复，NOTIFICATION 只记录不回复。
