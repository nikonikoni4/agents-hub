# 前后端实时消息推送方案

## 问题描述

在多Agent群聊场景中，消息的产生有两种来源：

1. **用户消息**：前端发起，后端处理后返回（传统HTTP请求-响应模式）
2. **Agent消息**：后端自主产生，前端被动接收

第二种场景是核心难点——多个Agent在后端自行交流，前端无法预知何时有新消息，需要一种机制来实时获取。

### 具体场景

```
用户A(前端) ─┐
Agent B ────┼→ 群聊 ─→ 前端需要实时显示所有消息
Agent C ────┤
Agent D ────┘
```

## 需求

1. **实时性**：Agent产生的消息需要在秒级内推送到前端
2. **多端支持**：桌面端(Electron)和移动端都需要能接收消息
3. **本地数据**：消息数据存储在桌面端本机，移动端通过局域网同步
4. **多端同步**：多个客户端看到的消息保持一致

## 技术方案对比

### 方案一：轮询 (Polling)

前端定时向后端请求新消息。

```
前端 ──GET /messages?after=123──→ 后端
前端 ──GET /messages?after=123──→ 后端
前端 ──GET /messages?after=123──→ 后端
...每秒一次...
```

**优点**：
- 实现最简单
- 兼容性好，无特殊协议要求

**缺点**：
- 延迟高（取决于轮询间隔）
- 浪费资源（大部分请求返回空）
- 服务端压力大（每个客户端每秒一个请求）

**不选原因**：实时性差，资源浪费严重。

### 方案二：长轮询 (Long Polling)

前端发起请求，后端hold住连接直到有新消息才返回。

```
前端 ──GET /messages──→ 后端
                        (等待...)
                        (有新消息!)
前端 ←── 返回消息 ──── 后端
前端 ──GET /messages──→ 后端
                        (等待...)
```

**优点**：
- 比普通轮询实时性好
- 兼容性好

**缺点**：
- 每次消息后需要重新建立连接
- 服务端需要维护大量挂起的连接
- 实现复杂度高于轮询

**不选原因**：相比WebSocket没有优势，且实现复杂。

### 方案三：SSE (Server-Sent Events)

服务端单向推送流，前端监听。

```
前端 ──GET /stream──→ 后端
前端 ←── 消息1 ────── 后端
前端 ←── 消息2 ────── 后端
前端 ←── 消息3 ────── 后端
(连接持续保持)
```

**优点**：
- 实现简单（比WebSocket简单）
- 自动重连
- 基于HTTP，兼容性好

**缺点**：
- 单向通信（前端发消息仍需普通HTTP请求）
- 不支持二进制数据

**可选场景**：如果只是简单的消息推送，SSE是不错的选择。但考虑到可能需要双向通信（如心跳、状态同步），WebSocket更灵活。

### 方案四：WebSocket

全双工持久连接，双方都能随时发消息。

```
前端 ════ WebSocket连接 ════ 后端
      ←── 消息推送 ──────────
      ──── 发送消息 ────────→
      ←── Agent消息推送 ─────
      ←── Agent消息推送 ─────
(连接持续保持，双方随时通信)
```

**优点**：
- 实时性最好
- 全双工通信
- 低延迟、低开销
- 支持二进制数据

**缺点**：
- 需要维护连接状态
- 某些代理/防火墙可能不支持
- 实现复杂度略高

**选择原因**：实时性最好，双向通信灵活，FastAPI和Electron都原生支持。

## 方案选择：WebSocket

### 选择理由

| 维度 | 轮询 | 长轮询 | SSE | WebSocket |
|------|------|--------|-----|-----------|
| 实时性 | 差 | 中 | 好 | 最好 |
| 资源消耗 | 高 | 中 | 低 | 低 |
| 双向通信 | 否 | 否 | 否 | 是 |
| 实现复杂度 | 低 | 中 | 低 | 中 |
| 断线重连 | 需自己实现 | 需自己实现 | 内置 | 需自己实现 |

WebSocket在实时性和灵活性上最优，且FastAPI原生支持，实现复杂度可接受。

## 核心逻辑设计

### 1. 消息流转逻辑

```
消息产生（用户/Agent）
    ↓
写入消息存储（SQLite）
    ↓
触发广播事件
    ↓
WebSocket推送给所有在线客户端
```

关键点：所有消息（无论来源）都走同一条路径，保证顺序一致。

### 2. 连接管理逻辑

后端维护一个连接池，记录每个WebSocket连接属于哪个房间：

```
连接池 = {
    room_1: [连接A, 连接B],
    room_2: [连接C],
    ...
}
```

广播时只推送给对应房间的连接。

### 3. 多端同步逻辑

```
桌面端（主设备）
    │
    ├── 本地前端：直接通过WebSocket连接
    │
    └── 手机端：通过局域网WebSocket连接
            │
            └── 扫码配对时获取桌面端IP和端口
```

同步机制：
- 新消息：通过WebSocket实时推送
- 断线重连：客户端带上 `last_message_id`，服务端补发缺失消息
- 历史消息：首次连接时拉取最近N条消息

### 4. 手机端配对逻辑

```
1. 桌面端生成配对QR码（包含局域网IP + 一次性token）
2. 手机扫码，获取连接地址
3. 手机发起WebSocket连接，携带token
4. 桌面端验证token，建立连接
5. 开始同步消息
```

## 技术栈

| 组件 | 技术选择 |
|------|---------|
| 后端框架 | FastAPI |
| WebSocket | FastAPI原生支持 |
| 消息存储 | SQLite（本地） |
| 桌面前端 | React + Electron |
| 移动端 | React Native 或 移动端Web |
| QR码生成 | Python qrcode库 |
