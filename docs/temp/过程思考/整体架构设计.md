# 后端代码架构和功能
1. agent_bridge 提供统一的调用agent的接口，只有调用功能，不包括会话维护，不同角色配置维护等，仅仅作为一个最基础的调用工具
2. agents 设置role的相关配置，主要作用是
    初始化：
        1. 动态注册role， 2. 每个role会在LOCAL_DATA_PATH = "local_data" 下创建一个agents以role为名称的文件夹 具体为 `LOCAL_DATA_PATH / agents / <role_name>` 3. 核心功能是配置claude code / codex 的全局目录文件 4. 默认claude 平台的角色复制 `Path.home() / .claude `的settings.json ，创建空白的CLAUDE.md 5. codex 从`Path.home() / .codex ` 复制 auth.json 、 config.toml和 rules文件夹 
        6. 配置角色metadata：
            ```json
            {
                "name": "角色名称",
                "type": "leader|team_member",
                "scope": [""], # 所属群聊
                "avatar": "/path/to/avatar.png",
                "abilities": ""
            }
            ```
    修改配置
        1. 增加skill，利用skill_manager迁移新的skill到 `local_data / agents / <role> / work_root/  skills`
        2. 修改settings.json 或 config.toml 
        3. 修改settings.json 或 config.toml 的部分内容的快捷方式，主要是为前端快速设置提供方法
            1. 对于claude code：修改permission模式，修改allow,deny ask
            2. 对于codex
        4. 选择头像，上传头像
        5. 返回结构化数据： role_name,ability,prompt
    未确认问题：
        聊天（单聊，群聊）是基于一个项目path的还是可以跨项目执行的？不然不用角色所属不同仓库，这样角色
3. skill_manager 主要功能，管理skill文件，主要位置： `local_data_path / skills`
        1. 预设一些skill文件，注意role的skill不属于skill_manager的职能范围
        2. 讲`local_data_path / skills`复制给各个role，前端表现为在role的配置界面添加skill
        3. 提供网络搜索skills并下载的功能[p2]，暂时不是实现

# local_data文件数据
1. agents： 存放agents - agent_role管理
    / assert  : 存放一些预设头像
    / <role> / work_root : 作为CODEX_HOME等， 存放skill，存放规则
    / <role> / avatar / current.png history_##.png
    / <role> / role.json : 存放角色元信息，包括，名称，角色的类型（leader / team_member），范围，头像，能力标签
2.  skills: 存放skill - skill_manager 管理
