# 秘书与GroupChatContext架构方案

## 设计日期
2026-05-27

## 讨论背景

### 原始问题

参考：[agentshub多智能体架构设计思考](./agentshub多智能体架构设计思考.md)

在设计agents-hub的多agent架构时，我们面临以下核心问题：

**问题1：需要支持的多agent模式**
1. workflow：按照设定顺序执行
2. orchestrate-subagent：subagent之间不能直接对话，必须通过主agent协调
3. subagent之间的对话：A委派B执行任务，不经过主agent
4. boss直接与subagent对话：@subagent机制

**问题2：subagent之间的对话如何处理？**

原始思考中提出了两种方式：
- 方式1：subagent A → main agent → subagent B
- 方式2：subagent A → 直接发送 → subagent B

核心疑问：
> "如果subagent之间的消息不经过main agent会怎么样？main agent可能会不知道进度，但是如果只是任务过程中的中间会话，那么main agent就不需要知道。"

**问题3：如何判断agent的结果是否应该返回到群聊记录？**

参考AutoGen的做法：默认都返回结果到聊天记录（buffer）

但这会导致上下文污染：
> "前端设计在群聊，但后端数据库讨论内容与前端完全无关，根本就不需要他知道这些上下文。"

**问题4：是否需要一个路由（秘书）？**

原始思考：
> "是否需要一个专门的路由（秘书），他判断subagent之间的内容以及结果是否需要发送给main agent和群聊，这个路由也是一个llm"

**核心洞察**：
> "本质上多agent架构设计是最优化不同agent之间的上下文"

### 讨论过程中确认的问题

**确认1：秘书的真正作用**

最初理解：秘书是"路由 + 信息压缩"
修正理解：秘书是"公共记忆提取器"

秘书的核心职责不只是路由（判断消息该不该进群聊），更重要的是**信息压缩和提取**：
- 从群聊的大量对话中提取关键信息
- 为不同agent生成定制化的上下文
- 避免AutoGen式的上下文污染

**确认2：A2A私聊是否进群聊？**

初步倾向：所有消息都进群聊，通过秘书压缩
疑问：这样路由是否还重要？

讨论结果：
- 所有A2A对话都应该进群聊（信息透明 > 效率优化）
- 路由的作用简化为"消息分发"，不做价值判断
- 秘书负责延迟压缩，而不是实时过滤

**确认3：群聊记录的生命周期**

- 原始记录：完整保留，可归档
- 压缩版本：版本化管理，支持增量更新
- 每个agent维护"已读版本"，只接收增量

**确认4：秘书的触发时机**

不是固定N条消息，而是基于token估算：
```
if estimate_tokens(recent_messages) > TOKEN_LIMIT:
    trigger_compression()
```

## 核心设计理念

**核心原则**：所有agent间的对话都进入群聊，通过"秘书"（公共记忆提取器）进行延迟压缩，每个agent只接收增量压缩版本。

**关键洞察**：
1. 多agent架构的本质是**最优化不同agent之间的上下文**
2. 秘书的作用是**公共记忆提取**，而不只是路由或信息压缩
3. 信息透明 > 效率优化（决策的可追溯性比消息数量优化更重要）

## 整体架构

```
┌─────────────────────────────────────────────────┐
│            GroupChatContext（群聊上下文）         │
│  ┌──────────────────────────────────────────┐   │
│  │  原始消息流（完整保留，可归档）            │   │
│  │  [msg1, msg2, msg3, ..., msgN]           │   │
│  └──────────────────────────────────────────┘   │
│                      ↓                           │
│  ┌──────────────────────────────────────────┐   │
│  │  秘书（Secretary）                        │   │
│  │  - 监控token数                            │   │
│  │  - 达到阈值触发压缩                       │   │
│  │  - 生成版本化的压缩记录                   │   │
│  └──────────────────────────────────────────┘   │
│                      ↓                           │
│  ┌──────────────────────────────────────────┐   │
│  │  压缩记录（版本化）                       │   │
│  │  v1: "前端完成登录页面"                   │   │
│  │  v2: "后端完成API接口"                    │   │
│  │  v3: "前端和后端确定了数据格式"           │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
                      ↓
        ┌─────────────┴─────────────┐
        ↓                           ↓
   Agent A                      Agent B
   (last_seen: v1)             (last_seen: v2)
   接收: v2, v3                接收: v3
```

## 三个核心组件

### 1. GroupChatContext（群聊上下文）

**职责**：
- 维护群聊的完整原始记录
- 管理压缩版本的历史
- 追踪每个agent的"已读版本"
- 提供增量上下文给新启动的agent

**核心数据结构**：
```
GroupChatContext {
    // 原始消息流
    raw_messages: [Message]
    
    // 压缩版本历史
    compressed_versions: [
        {version: 1, content: "...", token_range: [0, 1000]},
        {version: 2, content: "...", token_range: [1001, 2500]},
        ...
    ]
    
    // Agent版本追踪
    agent_versions: {
        "frontend_agent": {last_seen: 1, current: 3},
        "backend_agent": {last_seen: 2, current: 3},
        ...
    }
}
```

**关键方法**：
- `append(message)`: 添加新消息到原始记录
- `get_context_for_agent(agent_id)`: 获取agent需要的增量上下文
- `update_agent_version(agent_id, version)`: 更新agent的已读版本

### 2. Secretary（秘书 - 公共记忆提取器）

**职责**：
- 监控原始消息的token数
- 达到阈值时触发压缩
- 从对话中提取结构化记忆
- 生成版本化的压缩记录

**触发机制**：
- **不是固定N条消息**，而是基于token估算
- 当 `estimate_tokens(recent_messages) > TOKEN_LIMIT` 时触发
- TOKEN_LIMIT建议值：2000-3000 tokens

**压缩内容**：
```
压缩记录 {
    version: 版本号
    summary: "一句话总结这段对话"
    key_decisions: [
        {decision: "...", reason: "...", participants: ["agent1", "agent2"]}
    ]
    deliverables: ["文件路径1", "文件路径2"]
    dependencies: [
        {from: "frontend_agent", to: "backend_agent", what: "需要API接口"}
    ]
    interfaces: [
        {type: "api", endpoint: "/api/login", description: "..."}
    ]
    related_domains: ["frontend", "backend"]
}
```

**压缩策略**：
- 过滤噪音：纯信息查询（"XX文件在哪？"）在压缩时自动过滤
- 提取关键：只保留决策、产出、依赖关系
- 结构化：生成可索引的结构化数据

### 3. 消息分发机制（简化的路由）

**职责**：
- 消息寻址（找到目标agent的session）
- 消息类型标注（task/status/chat/query）
- 触发秘书检查

**核心流程**：
```
1. Agent发送消息
   ↓
2. 添加到GroupChatContext的原始记录
   ↓
3. 分发给目标agent（如果是@mention或广播）
   ↓
4. 通知Secretary检查是否需要压缩
```

**关键点**：
- 不判断"消息该不该进群聊"（全部进）
- 不判断"消息重不重要"（秘书延迟判断）
- 只负责"送达"，不负责"审查"

## 核心工作流程

### 场景1：Agent间协作对话

```
1. 前端agent @ 后端agent: "需要登录API"
   → 进入GroupChatContext原始记录
   → 分发给后端agent

2. 后端agent回复: "我来实现POST /api/login"
   → 进入GroupChatContext原始记录
   → 分发给前端agent

3. 前端和后端讨论了50轮API细节
   → 所有消息都进入原始记录
   → Secretary监控token数

4. Token数超过阈值（假设2000）
   → Secretary触发压缩
   → 生成v1压缩记录: "前端和后端确定了登录API格式：{...}"
   → 归档原始消息（可选）

5. 新的测试agent启动
   → 读取last_seen版本（假设为0）
   → 获取v1压缩记录
   → 测试agent知道"登录API已经确定"
```

### 场景2：Agent版本追踪

```
初始状态：
- 前端agent: last_seen=0, 参与了v1的讨论
- 后端agent: last_seen=0, 参与了v1的讨论
- 测试agent: 尚未启动

v1压缩完成后：
- 前端agent: last_seen=1（已经在session中，不需要接收v1）
- 后端agent: last_seen=1（已经在session中，不需要接收v1）
- 测试agent: last_seen=0（未启动）

测试agent启动时：
- 读取last_seen=0, current=1
- 接收v1压缩记录
- 更新last_seen=1

后续又产生了v2, v3压缩：
- 前端agent一直在线: last_seen=3（实时接收）
- 后端agent中途离线: last_seen=1
- 后端agent重新启动: 接收v2, v3增量
```

## 关键设计决策

### 决策1：A2A私聊是否进群聊？

**结论**：是，所有A2A对话都进群聊

**理由**：
1. **信息透明**：决策的可追溯性比消息数量优化更重要
2. **用户体验**：用户发的消息不能"被系统吞掉"
3. **完整历史**：未来可以回答"为什么做这个决策？"
4. **秘书过滤**：噪音由秘书在压缩时过滤，不需要实时判断

**反例**（如果不进群聊的问题）：
- 依赖agent"主动汇报"（不可靠）
- 决策历史不完整（私聊内容丢失）
- 无法追溯"为什么前端用了这个API格式？"
- 可能出现"暗箱操作"

### 决策2：秘书的触发时机

**结论**：基于token阈值的批量压缩

**触发条件**：
```
if estimate_tokens(recent_messages) > TOKEN_LIMIT:
    trigger_compression()
```

**不采用的方案**：
- ❌ 实时压缩（每条消息都处理）→ 成本太高
- ❌ 固定N条消息 → 不够灵活（有的消息长，有的短）
- ❌ 按需压缩（新agent启动时才压缩）→ 可能太晚

**防抖机制**：
- 30秒内的多次触发合并为一次
- 避免频繁调用LLM

### 决策3：路由的作用

**结论**：路由简化为"消息分发"，不做价值判断

**原来认为路由要做的**：
- ❌ 判断消息是否进群聊（不需要了，全部进）
- ❌ 判断消息是否需要压缩（秘书基于token阈值自动判断）
- ❌ 过滤无关消息（秘书压缩时自动过滤）

**现在路由只需要做**：
- ✅ 消息寻址（A → B，找到B的session）
- ✅ 消息类型标注（task/status/chat/query）
- ✅ 触发秘书检查（"是否达到压缩阈值？"）

**本质变化**：路由从"守门人"变成"邮递员"

### 决策4：群聊记录的生命周期

**短期记忆（当前任务）**：
- 原始消息完整保留
- 用于秘书压缩
- 用于用户查看完整历史

**中期记忆（压缩版本）**：
- 版本化的压缩记录
- 用于agent启动时的上下文注入
- 保留关键决策和产出

**长期记忆（未来考虑）**：
- 结构化知识（类似DeerFlow的facts）
- 跨会话的持久化记忆
- 当前方案聚焦在会话过程中，长期记忆暂不考虑

**归档策略**：
- 原始消息在压缩后可以归档（移出活跃内存）
- 压缩版本保留在GroupChatContext中
- 归档的原始消息仍可查询（用于审计和回溯）

## 与现有框架的对比

### vs AutoGen
- **AutoGen问题**：所有Worker共享buffer，上下文污染
- **我们的方案**：通过秘书压缩 + 版本追踪，每个agent只看增量

### vs MetaGPT
- **MetaGPT问题**：Memory无限增长，无截断策略
- **我们的方案**：基于token阈值触发压缩，控制上下文大小

### vs CrewAI
- **CrewAI方式**：只传递TaskOutput摘要，不保留完整历史
- **我们的方案**：保留完整原始记录 + 压缩版本，兼顾透明性和效率

### vs DeerFlow
- **DeerFlow方式**：Lead和SubAgent完全隔离，SubAgent无记忆
- **我们的方案**：所有agent共享群聊上下文，但通过版本追踪控制接收内容

## 优势总结

1. **信息透明**：所有对话进群聊，决策可追溯
2. **上下文优化**：通过版本追踪，agent只接收增量
3. **成本可控**：基于token阈值触发压缩，不是实时压缩
4. **用户体验好**：消息不会"消失"，都在群聊里
5. **架构简单**：路由简化为分发，秘书负责压缩，职责清晰

## 待解决的问题

1. **压缩质量**：如何保证秘书提取的信息足够准确？
2. **跨领域过滤**：如何让前端agent不看到无关的后端细节？（通过related_domains标签）
3. **实时性**：压缩是异步的，新agent启动时可能看到"过时"的压缩版本
4. **归档策略**：原始消息何时归档？归档后如何查询？
5. **错误处理**：如果秘书压缩失败怎么办？
