# MetaGPT
1. 消息传递使用发布-订阅模式，无统一的消息总线，每个agent一个私有消息队列
2. 在不同普通模式下，是一套比较灵活的SOP/workflow，每个agent干活不需要路由！而是由预先定义的“上一个动作”来触发下一个agent干活。[我没有深入了解]：但是如果预先定义的动作出现重叠，比如有两个agent都对某一个动作A感兴趣（动作A在他们的_watch监控列表里面），他们都会执行操作。这个不算缺点，有位有些时候可能需要某个动作A之后其他两个agent也需要执行。但是设计容易出错。可能需要一个专门的图了表示激活关系，避免出错。[可能的问题，我没有深入了解这部分]:目前没有了解如果某个agent出错无法完成任务，该模式的错误处理是什么，如果出现错误，但是由于是这个模式是由case_by和_watch事件驱动的流程，主agent，TL他是不知道任务进展的，出错了是否需要他进行调度？
普通模式虽然在结构上属于`docs\temp\研究报告\多智能体\多智能体协调模式.html`中所说的消息-订阅机制，但是他是由上一个动作确定性的指向下一个agent，解决了llm路由不确定性问题。但是流程追踪和调试性在一些可能的复杂的指向中会显得很麻烦，而且也不好设置。更多的可能还是一个比较简单或者小范围循环，跳转的workflow
3. 在MXG模式下，所有subagent的消息都有都会默认传给TL，由TL统一调度和路由任务。这个模式又回到了编排-子agent模式。
4. 最大的特点在于Action，tool的一种。内部包含了一个完整的动作，可能是llm_call，内部封装了一个节点的prompt。实现了节点的可复用性，一定程度上Action的组合比直接llm决定做什么确定性更高。
5. [可能的场景]： 我认为MetaGPT比较适合那种流程复用性高的工作，确定性较高但是想要一定工作的灵活性。简单模式可以进行Action拼接。他属于我之前思考的`docs\temp\过程思考\多智能体协调模式思考.md`中的无记忆的执行者。尽管他是有记忆模块，但是他的记忆是属于工作过程记录，无领域专业知识沉淀。如果改进记忆模块和prompt设置[prompt设置我没有深入查看]，增加以下外接的对于这个agent专业知识，把他当作Agent Team模式。
6. A2A方式：私有队列@执行

# deer flow
1. middleware中间件设计，类比于claude code 的hook。内置了多个安全措施、任务执行配置、错误处理的可复用中间件。
2. docker隔离，对于任务的安全性更高，而且每个子agent操作不同的实体，起到了隔离作用
3. 使用langgraph的单节点做状态保存，中断，人工介入等
4. [多agent架构]：典型的编排-子代理模式，子代理无记忆（长期），并且子代理之间不能进行沟通，只能由主代理安排
5. A2A方式：工具调用
