# 多智能体协调模式思考

> 日期：2026-05-23
> 来源：阅读《Multi-agent coordination patterns: Five approaches and when to use them》后的讨论

## 核心概念

### 五种协调模式

| 模式 | 适用场景 | 特点 |
|------|---------|------|
| Generator-Verifier | 质量敏感的输出，有明确评估标准 | 生成-验证循环 |
| Orchestrator-Subagent | 任务分解清晰，子任务有边界 | 编排者派发，子代理返回即销毁 |
| Agent Teams | 并行、独立、长时间运行的子任务 | 工人持久化，积累上下文 |
| Message Bus | 事件驱动流水线，代理生态增长 | 发布-订阅，松耦合 |
| Shared State | 协作式工作，代理相互构建发现 | 共享存储，无中心协调 |

## Orchestrator-Subagent vs Agent Teams

### 核心区别：生命周期

| 维度 | Orchestrator-Subagent | Agent Team |
|------|----------------------|------------|
| **生命周期** | 任务完成即销毁 | 长期存活 |
| **记忆** | 无/低，不跨任务积累 | 有，长期积累领域知识 |
| **划分维度** | 按任务（做什么） | 按领域实体（管什么） |
| **任务关联度** | 低（任务之间独立） | 高（围绕同一实体） |

### 关键洞察

> Agent Team 的核心在于任务分配是针对某个**实体**的，而不针对某个**任务**。
> 工人是"专家"，长期深耕某个领域，越干越熟练。

## 混合架构设计

### 场景：开发新功能

```
需求：开发某个功能
  → 可能涉及多个旧模块的改动
  → 新功能开发用任务型工人（Orchestrator-Subagent）
  → 旧模块改动用领域专家（Agent Team）
```

### 三层代理架构

```
第一层：主代理（Orchestrator）
  - 拆解任务、协调流程、综合结果
  - 不关心具体实现，只关心流程

第二层：维护者（Domain Expert）
  - 划分维度：按领域实体（用户模块、订单模块、支付模块）
  - 有记忆，熟悉模块历史
  - 知道边界条件、踩坑经验
  - 是"守门人"：知道什么能改、什么不能改

第三层：执行者（Task Worker）
  - 划分维度：按任务模式（代码执行、测试、审核）
  - 无记忆，任务完成即销毁
  - 是"工具人"：拿到任务就执行
```

### 路由判断

需要一个路由来判断是否需要领域专家介入：

| 维度 | 简单修改 | 复杂修改（需专家） |
|------|---------|-------------------|
| **影响范围** | 单文件/单模块 | 跨模块/跨服务 |
| **耦合度** | 无外部依赖 | 有其他模块依赖 |
| **历史踩坑** | 无相关记录 | 有过类似改动出过问题 |
| **业务关键性** | 普通字段 | 核心业务逻辑 |

### 三层协作示例

**需求："开发退款功能，涉及用户、订单、支付三个模块"**

```
主代理（第一层）
  ├─ PRD师（第三层执行者）→ 输出PRD
  ├─ 架构师（第三层执行者）→ 输出架构设计
  │
  ├─ 用户模块维护者（第二层）
  │    └─ "退款会影响用户信用积分"
  │         └─ 代码执行者（第三层）→ 实现积分扣减
  │
  ├─ 订单模块维护者（第二层）
  │    └─ "退款后订单状态要改为'已退款'"
  │         └─ 代码执行者（第三层）→ 实现状态流转
  │
  ├─ 支付模块维护者（第二层）
  │    └─ "退款要调用支付通道的退款接口"
  │         └─ 代码执行者（第三层）→ 实现退款调用
  │
  └─ 审核执行者（第三层）→ 整体代码审核
```

## 各层回答的问题

| 层级 | 问题 | 答案 |
|------|------|------|
| 第一层主代理 | 任务怎么拆？ | 流程拆解 |
| 第二层维护者 | 这个模块能不能改？ | 领域知识判断 |
| 第三层执行者 | 具体怎么实现？ | 任务模式执行 |
