## 1. 并发写入冲突

### 场景 1：多个 agent 同时回复群聊

```
Agent A 执行完成 → add_message() → save_group_chat_session()
Agent B 执行完成 → add_message() → save_group_chat_session()
```

**问题**：
- 如果 A 和 B 同时调用 `save_group_chat_session()`，后者会覆盖前者的写入
- 导致 A 的消息丢失

**影响文件**：
- `<group_chat_id>.jsonl`：消息历史
- `agent_session_state.json`：agent 状态

**解决方案（待实现）**：
1. **方案 A：使用 `asyncio.Lock`**
   ```python
   class GroupChatContext:
       def __init__(self, ...):
           self._save_lock = asyncio.Lock()
       
       async def save_group_chat_session(self):
           async with self._save_lock:
               # 写入文件操作
   ```
   - 优点：简单直接
   - 缺点：锁粒度大，可能影响性能

2. **方案 B：使用队列 + 单独的写入任务**
   ```python
   class GroupChatContext:
       def __init__(self, ...):
           self._save_queue = asyncio.Queue()
           self._save_task = asyncio.create_task(self._save_worker())
       
       async def _save_worker(self):
           while True:
               save_request = await self._save_queue.get()
               # 执行写入操作
       
       async def save_group_chat_session(self):
           await self._save_queue.put(SaveRequest(...))
   ```
   - 优点：串行化写操作，避免冲突
   - 缺点：实现复杂，需要管理后台任务

3. **方案 C：使用文件锁（跨进程）**
   ```python
   import fcntl  # Unix
   # 或 msvcrt  # Windows
   
   with open(file, 'r+') as f:
       fcntl.flock(f, fcntl.LOCK_EX)
       # 读写操作
       fcntl.flock(f, fcntl.LOCK_UN)
   ```
   - 优点：支持跨进程
   - 缺点：平台相关，Windows 和 Unix 实现不同

**推荐方案**：
- 短期：方案 A（`asyncio.Lock`），简单快速
- 长期：方案 B（队列），更好的性能和可扩展性

## 2. 压缩时有新消息到达

### 场景 2：压缩过程中有新消息

```
压缩线程：读取 messages[last_compact_loc:] → 调用 LLM 压缩 → 更新 last_compact_loc
Agent A：add_message() → save_group_chat_session()
```

**问题**：
- 压缩线程读取消息时，`last_compact_loc = 50`，消息总数 = 100
- 压缩过程中（LLM 调用需要几秒），Agent A 添加了新消息，消息总数 = 101
- 压缩完成后，更新 `last_compact_loc = 100`
- 但此时消息总数 = 101，第 101 条消息会被遗漏（下次压缩会从 100 开始）

**边界情况**：
1. **压缩过程中添加多条消息**：
   - 压缩开始：`last_compact_loc = 50`，总数 = 100
   - 压缩过程中：添加了 5 条消息，总数 = 105
   - 压缩完成：更新 `last_compact_loc = 100`
   - 结果：第 101~105 条消息会在下次压缩时被处理（正确）

2. **压缩过程中触发第二次压缩**：
   - 第一次压缩：处理 50~100
   - 第二次压缩：处理 100~150（假设又有 50 条新消息）
   - 如果第一次压缩还没完成，第二次压缩就开始了，会有冲突

**解决方案（待实现）**：

### 方案 A：记录压缩范围（推荐）

```python
async def compact_messages(self, agent_info: dict[str, str]):
    # 1. 记录压缩范围
    compact_from = self.group_chat_session.last_compacted_loc
    compact_to = len(self.group_chat_session.messages)
    
    # 2. 获取要压缩的消息
    messages_to_compact = self.group_chat_session.messages[compact_from:compact_to]
    
    # 3. 调用 LLM 压缩（可能需要几秒）
    compact_result = await llm_call.execute(compact_prompt)
    
    # 4. 保存压缩结果
    save_compact_history(compact_result)
    
    # 5. 更新 last_compacted_loc（使用记录的 compact_to，而不是当前的 len(messages)）
    self.group_chat_session.last_compacted_loc = compact_to
    self.save_group_chat_session()
```

**优点**：
- 简单，不需要锁
- 即使压缩过程中有新消息，也不会遗漏

**缺点**：
- 如果压缩过程中添加了很多消息，下次压缩的范围会很大

### 方案 B：压缩时加锁

```python
async def compact_messages(self, agent_info: dict[str, str]):
    async with self._compact_lock:
        # 压缩过程中，禁止添加新消息
        messages_to_compact = self.group_chat_session.get_uncompact_messages()
        compact_result = await llm_call.execute(compact_prompt)
        save_compact_history(compact_result)
        self.group_chat_session.last_compacted_loc = len(self.group_chat_session.messages)
```

**优点**：
- 保证压缩过程中不会有新消息

**缺点**：
- 压缩过程中（可能需要几秒），所有 agent 都无法添加消息
- 影响系统响应性

### 方案 C：压缩标志位

```python
class GroupChatSession:
    is_compacting: bool = False  # 是否正在压缩
    
async def compact_messages(self, agent_info: dict[str, str]):
    if self.group_chat_session.is_compacting:
        print("压缩正在进行中，跳过本次压缩")
        return
    
    self.group_chat_session.is_compacting = True
    try:
        # 压缩逻辑（使用方案 A 的记录范围方式）
        compact_from = self.group_chat_session.last_compacted_loc
        compact_to = len(self.group_chat_session.messages)
        # ...
        self.group_chat_session.last_compacted_loc = compact_to
    finally:
        self.group_chat_session.is_compacting = False
```

**优点**：
- 避免重复压缩
- 不阻塞消息添加

**缺点**：
- 需要维护标志位

**推荐方案**：
- **方案 A + 方案 C**：记录压缩范围 + 压缩标志位

## 3. 多个 agent 同时加载上下文

### 场景 3：并发加载上下文

```
Agent A：get_context() → 更新 agent_state
Agent B：get_context() → 更新 agent_state
```

**问题**：
- 如果 A 和 B 同时调用 `get_context()`，都会更新 `agent_session_state.json`
- 后者会覆盖前者的写入

**影响文件**：
- `agent_session_state.json`

**解决方案（待实现）**：

### 方案 A：使用 `asyncio.Lock`

```python
class GroupChatContext:
    def __init__(self, ...):
        self._agent_state_lock = asyncio.Lock()
    
    async def save_agent_session_id(self):
        async with self._agent_state_lock:
            # 写入文件操作
```

### 方案 B：每个 agent 独立文件

```
agent_session_state/
  ├── agent_A.json
  ├── agent_B.json
  └── agent_C.json
```

**优点**：
- 每个 agent 的状态独立，不会冲突

**缺点**：
- 文件数量增加
- 需要修改加载逻辑

**推荐方案**：
- **方案 A**：使用锁，简单直接

## 4. 长度计算的边界情况

### 场景 4：多个 agent 同时回复，长度计算错误

```
Agent A：add_message() → len(messages) = 100
Agent B：add_message() → len(messages) = 101
Agent A：save_group_chat_session() → 写入 100 条消息
Agent B：save_group_chat_session() → 写入 101 条消息（覆盖 A 的写入）
```

**问题**：
- A 的消息丢失

**解决方案**：
- 使用方案 1 的锁或队列，确保 `add_message()` 和 `save_group_chat_session()` 是原子操作

**问题**：
- 谁来调用 `compact_messages()`？
- 什么时候调用？

**可能的触发时机**：

1. **每次添加消息后检查**：
   ```python
   async def add_message(self, agent_result):
       self.group_chat_session.add_message(agent_result)
       
       # 检查是否需要压缩
       if self._should_compact():
           await self.compact_messages(agent_info)
   ```
   - 优点：及时压缩
   - 缺点：频繁检查，可能影响性能

2. **定期检查（后台任务）**：
   ```python
   async def _compact_worker(self):
       while True:
           await asyncio.sleep(60)  # 每分钟检查一次
           if self._should_compact():
               await self.compact_messages(agent_info)
   ```
   - 优点：不影响消息添加的性能
   - 缺点：可能延迟压缩

3. **手动触发**：
   - 由 GroupChat 或 orchestration 层决定何时压缩
   - 优点：灵活
   - 缺点：需要上层管理

**推荐方案**：
- **方案 2**：定期检查（后台任务）
