# 并发和压缩问题

## 问题概述

在实现 Context 层时，发现了多个并发和压缩相关的问题。

**✅ 已解决**：通过拆分 `GroupChatRepository`（持久化层 + 并发控制）和 `GroupChatContext`（业务逻辑层），使用 `asyncio.Lock` 保护文件读写操作。

**⚠️ 待验证**：需要编写测试用例验证并发场景。

---

## 架构设计

### 职责分离

```
GroupChatRepository (持久化层)
  ├── 文件读写（GroupChatSession, agent_session_state, compact_history）
  ├── 并发控制（3 个锁）
  │   ├── _session_lock: 保护 group_chat_session 文件
  │   ├── _agent_state_lock: 保护 agent_session_state 文件
  │   └── _compact_lock: 保护 compact_history 文件

GroupChatContext (业务逻辑层)
  ├── 消息管理（add_message）
  ├── Session 管理（update_agent_session_id）
  └── 压缩管理（compact_messages）

AgentContext (Agent 上下文)
  └── 增量加载（get_context）
```

### 并发控制策略

1. **读操作无锁**：`load_*()` 方法不加锁，允许并发读取
2. **写操作加锁**：`save_*()` 方法使用 `asyncio.Lock` 保护
3. **锁粒度**：每个文件独立的锁，避免不必要的阻塞

## 1. 并发写入冲突（✅ 已解决）

### 场景 1：多个 agent 同时回复群聊

```
Agent A 执行完成 → add_message() → save_group_chat_session()
Agent B 执行完成 → add_message() → save_group_chat_session()
```

**问题**：
- 如果 A 和 B 同时调用 `save_group_chat_session()`，后者会覆盖前者的写入
- 导致 A 的消息丢失

**影响文件**：
- `<group_chat_id>.jsonl`：消息历史
- `agent_session_state.json`：agent 状态

**✅ 解决方案**：

在 `GroupChatRepository` 中使用 `asyncio.Lock` 保护文件写入：

```python
class GroupChatRepository:
    def __init__(self, ...):
        self._session_lock = asyncio.Lock()  # 保护 group_chat_session 文件
        self._agent_state_lock = asyncio.Lock()  # 保护 agent_session_state 文件
    
    async def save_group_chat_session(self, session: GroupChatSession):
        async with self._session_lock:
            # 写入文件操作
    
    async def save_agent_session_state(self, state: dict[str, AgentSessionInfo]):
        async with self._agent_state_lock:
            # 写入文件操作
```

**优点**：
- 简单直接，易于理解
- 锁粒度合理（每个文件独立的锁）
- 不影响读操作性能

## 2. 压缩时有新消息到达（⚠️ 待实现）

### 场景 2：压缩过程中有新消息

```
压缩线程：读取 messages[last_compact_loc:] → 调用 LLM 压缩 → 更新 last_compact_loc
Agent A：add_message() → save_group_chat_session()
```

**问题**：
- 压缩线程读取消息时，`last_compact_loc = 50`，消息总数 = 100
- 压缩过程中（LLM 调用需要几秒），Agent A 添加了新消息，消息总数 = 101
- 压缩完成后，更新 `last_compact_loc = 100`
- 但此时消息总数 = 101，第 101 条消息会被遗漏（下次压缩会从 100 开始）

**边界情况**：
1. **压缩过程中添加多条消息**：
   - 压缩开始：`last_compact_loc = 50`，总数 = 100
   - 压缩过程中：添加了 5 条消息，总数 = 105
   - 压缩完成：更新 `last_compact_loc = 100`
   - 结果：第 101~105 条消息会在下次压缩时被处理（正确）

2. **压缩过程中触发第二次压缩**：
   - 第一次压缩：处理 50~100
   - 第二次压缩：处理 100~150（假设又有 50 条新消息）
   - 如果第一次压缩还没完成，第二次压缩就开始了，会有冲突

**⚠️ 推荐方案（待实现）**：

### 方案 A：记录压缩范围 + 压缩标志位

```python
class GroupChatSession:
    is_compacting: bool = False  # 是否正在压缩

async def compact_messages(self, agent_info: dict[str, str]):
    # 1. 检查是否正在压缩
    if self.group_chat_session.is_compacting:
        print("压缩正在进行中，跳过本次压缩")
        return
    
    self.group_chat_session.is_compacting = True
    try:
        # 2. 记录压缩范围
        compact_from = self.group_chat_session.last_compacted_loc
        compact_to = len(self.group_chat_session.messages)
        
        # 3. 获取要压缩的消息
        messages_to_compact = self.group_chat_session.messages[compact_from:compact_to]
        
        # 4. 调用 LLM 压缩（可能需要几秒）
        compact_result = await llm_call.execute(compact_prompt)
        
        # 5. 保存压缩结果
        await self.repository.save_compact_history(compact_result)
        
        # 6. 更新 last_compacted_loc（使用记录的 compact_to，而不是当前的 len(messages)）
        self.group_chat_session.last_compacted_loc = compact_to
        await self.repository.save_group_chat_session(self.group_chat_session)
    finally:
        self.group_chat_session.is_compacting = False
```

**优点**：
- 简单，不需要额外的锁
- 即使压缩过程中有新消息，也不会遗漏
- 避免重复压缩

**缺点**：
- 如果压缩过程中添加了很多消息，下次压缩的范围会很大

## 3. 多个 agent 同时加载上下文（✅ 已解决）

### 场景 3：并发加载上下文

```
Agent A：get_context() → 更新 agent_state
Agent B：get_context() → 更新 agent_state
```

**问题**：
- 如果 A 和 B 同时调用 `get_context()`，都会更新 `agent_session_state.json`
- 后者会覆盖前者的写入

**影响文件**：
- `agent_session_state.json`

**✅ 解决方案**：

在 `GroupChatRepository` 中使用 `asyncio.Lock` 保护 agent state 文件写入：

```python
class GroupChatRepository:
    def __init__(self, ...):
        self._agent_state_lock = asyncio.Lock()
    
    async def save_agent_session_state(self, state: dict[str, AgentSessionInfo]):
        async with self._agent_state_lock:
            # 写入文件操作
```

**优点**：
- 简单直接
- 保证并发安全

## 4. 长度计算的边界情况（✅ 已解决）

### 场景 4：多个 agent 同时回复，长度计算错误

```
Agent A：add_message() → len(messages) = 100
Agent B：add_message() → len(messages) = 101
Agent A：save_group_chat_session() → 写入 100 条消息
Agent B：save_group_chat_session() → 写入 101 条消息（覆盖 A 的写入）
```

**问题**：
- A 的消息丢失

**✅ 解决方案**：
- 使用 `_session_lock` 保护 `save_group_chat_session()`，确保 `add_message()` 和 `save_group_chat_session()` 的原子性
